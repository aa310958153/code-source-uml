📋 一、业务背景与挑战

1.1 行业趋势

随着大模型技术的普及，企业构建各类 AI 智能体应用时普遍面临以下共性问题：

❌ 缺乏统一交互入口

  • 各职能部门构建的 Agent 系统各自为政
  • 前端需对接多个独立服务，接口管理复杂
  • 维护成本高，重复建设严重

❌ 会话状态难以统一管理

  • 多轮对话场景下，不同 Agent 实例之间无法共享上下文
  • 用户体验割裂，交互连续性差

❌ 运维与监控能力薄弱

  • 缺少对各 Agent 服务的统一调用链追踪
  • 缺乏性能监控与异常告警机制

1.2 现有系统局限性

  • 通用性差：仅适用于单一场景，无法支持其他业务线快速接入
  • 重复开发成本高：每次新增智能体均需从零开发基础能力
  • 扩展性不足：专有性与定制化设计导致难以横向扩展

🎯 二、建设目标

🎯 本期目标

  • 构建统一 AI 接入层，屏蔽后端 Agent 差异性
  • 实现前端请求的统一接收、鉴权与路由分发
  • 支持多类型智能体快速接入（Dify 平台快搭、Spring AI 自研等）
  • 建立统一的会话上下文存储机制
  • 结合 LLM 实现用户意图识别与动态路由
  • 集成 RBAC 权限模型，保障平台安全性

🔮 中长期目标

  • 构建具备任务理解与自动编排能力的智能调度引擎
  • 实现用户复杂意图的自动识别、任务拆解
  • 支持多 Agent 协同执行

⚙️ 五、关键技术特性

5.1 会话管理方案

会话结构设计

  • Session ID：全局唯一标识
  • Scene ID：业务场景隔离
  • User ID：用户身份标识
  • Message List：最近 N 轮对话（滑动窗口）
  • System Prompt：场景化提示词

存储方案

  • MongoDB：持久化存储
  • Redis：缓存活跃会话
  • 滑动窗口：保留最近 5 轮
  • 摘要记忆：可选优化

5.2 权限控制策略

🔐 RBAC 权限模型

  • 角色定义：店员、店长、研发人员、数据分析师等
  • 权限粒度：基于场景+Agent 维度的访问控制
  • 实现方式：用户请求携带 appId 和 userId，根据角色查询授权范围，意图识别仅在授权范围内匹配
