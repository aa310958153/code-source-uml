一、核心洞察：工具即策略

一句话定义：工具链层是 Agent 的"双手" —— 每个工具不仅是一个功能单元，更是一个 Harness 策略。

Task 解决上下文污染问题，Todo 解决规划跑偏问题，Question 解决关键决策不确定性问题，
Bash 解决执行环境安全问题，Edit 解决代码修改精确性问题，Truncation 解决上下文溢出问题。

二、Tool.define：统一工具定义接口

所有工具通过 Tool.define() 统一声明，它自动提供参数校验、错误格式化和输出截断：

// tool.ts — Tool.define 核心结构
export namespace Tool {
    export interface Info {
        id: string
        init: (ctx?: InitContext) => Promise<{
            description: string           // 工具描述（来自 .txt 文件）
            parameters: ZodType           // Zod schema 参数定义
            execute(args, ctx): Promise<{
                title: string             // UI 展示标题
                metadata: Metadata        // 结构化元数据（diff、diagnostics等）
                output: string            // 文本输出（会被自动截断）
                attachments?: FilePart[]  // 可选的文件附件（图片等）
            }>
            formatValidationError?(error: ZodError): string  // 自定义校验错误
        }>
    }

    export function define(id, init): Info {
        return {
            id,
            init: async (initCtx) => {
                const toolInfo = ...
                toolInfo.execute = async (args, ctx) => {
                    // 1. 自动 Zod 参数校验
                    toolInfo.parameters.parse(args)
                    // 2. 执行实际逻辑
                    const result = await execute(args, ctx)
                    // 3. 自动 Truncation 截断
                    const truncated = await Truncate.output(result.output, {}, initCtx?.agent)
                    return { ...result, output: truncated.content }
                }
                return toolInfo
            },
        }
    }
}

// 每个工具执行时都能访问的上下文

    sessionID: string         // 当前会话
    messageID: string         // 当前消息

设计精妙之处：Tool.define() 用装饰器模式自动注入了三个横切关注点：
① Zod 校验 — 参数不合法时直接抛错，不会执行业务逻辑
② Truncation — 超长输出自动截断并持久化到文件
③ formatValidationError — 允许工具自定义错误提示（如 BatchTool 的友好格式）

五、Bash 工具：Tree-sitter 命令解析与权限双请求

Bash 是最复杂的工具。它在执行前通过 Tree-sitter 解析命令 AST，提取出所有子命令和路径参数：

5.1 BashArity：160+ 命令粒度字典

BashArity 将命令解析为"人类可理解的命令模式"，用于权限的 "always approve" 记忆：

// arity.ts — BashArity.prefix() 示例
"git checkout main"     → always = "git checkout *"  (git arity=2)
"npm run dev"           → always = "npm run dev *"    (npm run arity=3)
"rm -rf /tmp/foo"       → always = "rm *"             (rm arity=1)
"docker compose up"     → always = "docker compose up *"(docker compose arity=3)
"python script.py"      → always = "python script.py *"(default: 取全部)

5.2 路径参数解析

// bash.ts — 对 cd/rm/cp/mv/mkdir/touch/chmod/chown/cat 的路径解析
if (["cd", "rm", "cp", "mv", "mkdir", "touch", "chmod", "chown", "cat"].includes(command[0])) {
    for (const arg of command.slice(1)) {
        if (arg.startsWith("-")) continue  // 跳过 flag
        const resolved = await $`realpath ${arg}`.cwd(cwd).nothrow().text()
        if (resolved && !Instance.containsPath(resolved)) {
            directories.add(dir)  // 标记为外部目录，需要权限
        }
    }
}
为什么不直接用字符串匹配？Tree-sitter 能正确处理管道、重定向、子shell等复杂语法。
例如 cat foo.txt | grep "error" > out.log 会被解析为 3 个 command 节点， 每个都有独立的命令名和参数，确保权限检查不遗漏任何子命令。

六、Edit 工具：9 层模糊匹配替换策略

Edit 是 OpenCode 最精密的工具之一。当 LLM 提供的 oldString 不能精确匹配时，它会逐层尝试 9 种 Replacer 策略：

// edit.ts — 9 层 Replacer 瀑布
for (const replacer of [
    SimpleReplacer,              // ① 精确匹配
    LineTrimmedReplacer,         // ② 忽略行首尾空格
    BlockAnchorReplacer,         // ③ 首尾锚定 + Levenshtein
    WhitespaceNormalizedReplacer,// ④ 空白归一化
    IndentationFlexibleReplacer, // ⑤ 缩进灵活匹配
    EscapeNormalizedReplacer,    // ⑥ 转义字符还原
    TrimmedBoundaryReplacer,     // ⑦ 边界裁剪
    ContextAwareReplacer,        // ⑧ 上下文感知
    MultiOccurrenceReplacer,     // ⑨ 多次出现处理
]) {
    for (const search of replacer(content, oldString)) {
        const index = content.indexOf(search)
        if (index === -1) continue
        if (replaceAll) return content.replaceAll(search, newString)
        const lastIndex = content.lastIndexOf(search)
        if (index !== lastIndex) continue  // 多匹配 → 跳过，要求更精确
        return content.substring(0, index) + newString + content.substring(index + search.length)
    }
}
为什么需要 9 层策略？LLM 生成的 oldString 经常与原文有微小差异：多余的缩进、转义字符差异、 行尾空格等。逐层降级策略让 Edit 能在保持安全性的同时，最大化 LLM 编辑的成功率。
关键安全约束：当某层找到多个匹配时不会盲目替换，而是跳过（要求更精确的上下文），只有唯一匹配才执行。

七、Task 工具：子 Agent 上下文隔离

Task 工具创建独立的子 Session，让子 Agent 在自己的上下文中工作，不污染主对话：

// task.ts — 子 Agent 创建核心逻辑
const session = await Session.create({
    parentID: ctx.sessionID,              // 关联父 Session
    title: params.description + ` (@${agent.name} subagent)`,
    permission: [
        { permission: "todowrite", pattern: "*", action: "deny" },  // 子 Agent 禁止写 Todo
        { permission: "todoread",  pattern: "*", action: "deny" },  // 子 Agent 禁止读 Todo
        ...(hasTaskPermission ? [] : [
            { permission: "task", pattern: "*", action: "deny" },   // 无 task 权限则禁止嵌套
        ]),
    ],
})

// 在子 Session 中执行
const result = await SessionPrompt.prompt({
    sessionID: session.id,
    agent: agent.name,
    tools: { todowrite: false, todoread: false, ... },  // 显式禁用工具
    parts: promptParts,
})

// 返回结果 + task_id（用于恢复）
return { output: `task_id: ${session.id}\n\n\n${text}\n` }
上下文隔离的价值：
• 主 Agent 的上下文不会被大量搜索/读取操作膨胀
• 子 Agent 可以用不同的模型（explore 用轻量模型）
• task_id 允许后续恢复同一子 Session，实现"续接任务"
• 子 Agent 默认禁止 todowrite/todoread，避免修改主 Agent 的计划

八、Permission 集成：disabled 预过滤 + 运行时拦截

工具链与权限系统有两个交互点：

8.1 disabled()：工具注册阶段的预过滤

// permission/next.ts — disabled()
// 在 LLM.stream() 的 resolveTools() 中调用
const disabled = PermissionNext.disabled(Object.keys(input.tools), input.agent.permission)
// 全局 deny 的工具直接从工具列表中删除，LLM 根本看不到它们

export function disabled(tools: string[], ruleset: Ruleset): Set<string> {
    const result = new Set<string>()
    for (const tool of tools) {
        // edit/write/patch/multiedit 统一映射为 "edit" 权限
        const permission = EDIT_TOOLS.includes(tool) ? "edit" : tool
        const rule = ruleset.findLast((r) => Wildcard.match(permission, r.permission))
        if (rule && rule.pattern === "*" && rule.action === "deny") {
            result.add(tool)
        }
    }
    return result
}

8.2 ctx.ask()：工具执行时的运行时权限

// 每个工具在执行前调用 ctx.ask() 请求权限
// 例：Edit 工具
await ctx.ask({
    permission: "edit",                           // 权限类型
    patterns: [path.relative(worktree, filePath)], // 具体模式（文件路径）
    always: ["*"],                                 // "always approve" 的模式
    metadata: { filepath, diff },                  // UI 展示信息
})

// 例：Bash 工具（双重权限请求）
// 第一次：检查外部目录
await ctx.ask({ permission: "external_directory", patterns: globs, ... })
// 第二次：检查命令本身
await ctx.ask({ permission: "bash", patterns: [...], always: [...], ... })
两阶段权限控制的价值：
disabled() 是"静态"过滤 — 如果 Agent 配置了 edit: { *: deny }， 那 Edit/Write/ApplyPatch 工具不会出现在 LLM 的工具列表中，LLM 甚至不知道它们的存在。
ctx.ask() 是"动态"拦截 — 工具执行时根据具体的文件路径/命令内容请求权限， 用户可以逐次审批或"always approve"某类操作。

九、Truncation 系统：输出溢出保护

当工具输出超过 2000 行或 50KB 时，Truncation 系统自动截断并将完整内容持久化到文件：

export const MAX_BYTES = 50 * 1024  // 50KB

export async function output(text, options = {}, agent) {
    const lines = text.split("\n")
    const totalBytes = Buffer.byteLength(text, "utf-8")

    if (lines.length <= maxLines && totalBytes <= maxBytes) {
        return { content: text, truncated: false }
    }

    // 按方向（head/tail）截取
    // ... 截取逻辑 ...

    // 完整内容持久化到文件
    const filepath = path.join(DIR, Identifier.ascending("tool"))
    await Filesystem.write(filepath, text)

    // 根据 Agent 是否有 Task 工具，给出不同的建议

        ? `Use the Task tool to have explore agent process this file`

    return { content: `${preview}\n\n...${removed} ${unit} truncated...\n\n${hint}`,

十、Batch 工具：并行执行引擎

Batch 工具（实验性）允许 LLM 在一次调用中并行执行多个工具：

// batch.ts — 并行执行核心
const toolCalls = params.tool_calls.slice(0, 25)  // 最多 25 个
const discardedCalls = params.tool_calls.slice(25) // 超出的标记为错误

// 每个调用独立执行（包含自己的 UI part 状态更新）
const executeCall = async (call) => {
    // 1. 检查黑名单（batch 自身不能嵌套）
    if (DISALLOWED.has(call.tool)) throw new Error(...)

    // 2. 查找工具并校验参数
    const tool = toolMap.get(call.tool)
    const validatedParams = tool.parameters.parse(call.parameters)

    // 3. 更新 part 状态为 running
    await Session.updatePart({ state: { status: "running", ... } })

    // 4. 执行工具
    const result = await tool.execute(validatedParams, { ...ctx, callID: partID })

    // 5. 更新 part 状态为 completed/error
    await Session.updatePart({ state: { status: "completed", ... } })
}

// Promise.all 并行执行所有调用
const results = await Promise.all(toolCalls.map((call) => executeCall(call)))
Batch 的设计约束：
• DISALLOWED = ["batch"] — 禁止嵌套 Batch 调用
• 每个子调用都有独立的 part 状态追踪（running → completed/error）
• 超出 25 个上限的调用直接标记为错误，不会执行
• MCP 等外部工具不能 Batch — 只支持注册表内的内置工具

十二、设计优势总结

🔌

统一接口 + 自动横切

Tool.define() 自动注入 Zod 校验、Truncation 截断、错误格式化。开发者只需关注业务逻辑，声明参数 schema 即可。

🧠

工具即策略

每个工具都解决一个 Harness 问题：Task=上下文隔离，Todo=规划约束，Question=人在回路，Edit=精确修改，Bash=环境安全。

🌳

Tree-sitter 深度解析

Bash 工具不依赖字符串匹配，而是通过 Tree-sitter 解析命令 AST，精确提取子命令、路径参数，确保权限检查无遗漏。

🔄

9 层模糊匹配编辑

从精确匹配到 Levenshtein 相似度、从空白归一化到缩进灵活匹配，逐层降级最大化 LLM 编辑成功率。

🔒

双阶段权限控制

disabled() 在注册阶段过滤全局 deny 的工具（LLM 不可见），ctx.ask() 在执行阶段逐次请求权限（用户可审批/拒绝）。

📦

上下文隔离

Task 工具创建独立子 Session，子 Agent 在自己的上下文中工作。主对话不会因为大量搜索/读取而膨胀。

⚡

Batch 并行引擎

一次调用并行执行最多 25 个工具，每个调用有独立的 UI 状态追踪。禁止嵌套，外部工具不可 batch。

📏

自动溢出保护

Truncation 系统在 Tool.define 层自动生效，2000行/50KB 双重限制。完整输出持久化到文件，7天定时清理。

🎯

模型自适应

GPT-5 系列自动切换 apply_patch，Claude 使用 edit+write。WebSearch/CodeSearch 根据 Provider 条件开放。

🔧

三层扩展体系

内置工具（硬编码）+ 自定义工具（文件系统发现）+ Plugin 工具（插件注入），三种来源统一适配。

🛡️

BashArity 命令字典

160+ 命令的粒度映射，将 "npm run dev --port 3000" 抽象为 "npm run dev *"，让权限记忆既精确又宽泛。

🩺

LSP 诊断闭环

Edit/Write/ApplyPatch 执行后自动触发 LSP 诊断，将编译错误反馈给 Agent，形成"修改→检查→修复"闭环。

一句话总结：OpenCode 的工具链层是一个"声明式定义 + 自动横切 + 权限双控 + 智能降级"的工具体系。
它用 Tool.define() 统一接口抽象、用 9 层 Replacer 最大化编辑成功率、用 Tree-sitter 精确解析命令、
用 disabled() + ctx.ask() 实现双阶段权限控制、用 Task 实现上下文隔离、用 Truncation 防止溢出，
最终让每个工具都成为 Harness 策略的具体实现 —— 让 Agent 既有能力、又有边界。
