一、Session — 一切的容器

Session 是 OpenCode 中最核心的抽象。每一次用户与 AI 的对话，都发生在一个 Session 中。Session 承载了：

维度内容
消息历史所有 user/assistant 消息及其 parts（text、tool、file、reasoning...）
Agent 标识当前使用的 Agent（build/plan/...），每条消息可独立指定
模型信息使用的 provider 和 model ID
权限规则session.permission — 覆盖 Agent 默认权限
统计信息cost、tokens（input/output/reasoning/cache）、diff summary
快照追踪每步开始/结束时的 git tree hash，用于计算文件变更
父子关系parentID 关联，子 Agent Session 指向父 Session

四、ReAct 循环 — loop() 全景

loop() 是 OpenCode 的心脏。它实现了经典的 ReAct（Reasoning + Acting）循环：

五、循环入口：prompt() → loop()

// session/prompt.ts
export const prompt = fn(PromptInput, async (input) => {
    const session = await Session.get(input.sessionID)
    await SessionRevert.cleanup(session)      // 清理回退状态

    // 1. 创建 user 消息
    const message = await createUserMessage(input)
    await Session.touch(input.sessionID)      // 更新时间戳

    // 2. 向后兼容：tools 参数转为 permission
    const permissions: PermissionNext.Ruleset = []
    for (const [tool, enabled] of Object.entries(input.tools ?? {})) {
        permissions.push({ permission: tool, action: enabled ? "allow" : "deny", pattern: "*" })
    }

    // 3. noReply 模式：只创建消息，不启动循环
    if (input.noReply === true) return message

    // 4. 启动 ReAct 循环
    return loop({ sessionID: input.sessionID })
})
noReply 模式：某些场景下只需要创建用户消息（如添加上下文引用），不需要 AI 响应。此时 noReply: true 跳过循环。

六、消息流重建与 filterCompacted

每次循环迭代开始时，都需要从数据库重建完整的消息流：

// loop() 内部
let msgs = await MessageV2.filterCompacted(MessageV2.stream(sessionID))

// 从后向前扫描，找到关键消息
let lastUser, lastAssistant, lastFinished
let tasks = []  // 待处理的 compaction/subtask
for (let i = msgs.length - 1; i >= 0; i--) {
    const msg = msgs[i]
    if (!lastUser && msg.info.role === "user") lastUser = msg.info
    if (!lastAssistant && msg.info.role === "assistant") lastAssistant = msg.info
    if (!lastFinished && msg.info.role === "assistant" && msg.info.finish)
        lastFinished = msg.info
    // 收集待处理的任务（compaction 或 subtask）
    const task = msg.parts.filter(p => p.type === "compaction" || p.type === "subtask")
    if (task && !lastFinished) tasks.push(...task)
}

完成判断逻辑

// 如果最后一条 assistant 消息已完成（非 tool-calls），且它在最后一条 user 之后
if (
    lastAssistant?.finish &&
    !["tool-calls", "unknown"].includes(lastAssistant.finish) &&
    lastUser.id < lastAssistant.id
) {
    break  // 退出循环，AI 已完成响应
}

finish 字段的值来自 AI SDK 的 finishReason： "stop" = 正常结束，"tool-calls" = 需要执行工具后继续，"unknown" = 不确定。

七、step 计数与 maxSteps 保护

// loop() 中
let step = 0
while (true) {
    step++
    // ...
    const agent = await Agent.get(lastUser.agent)
    const maxSteps = agent.steps ?? Infinity    // Agent 配置的最大步数
    const isLastStep = step >= maxSteps

    // 最后一步注入 MAX_STEPS 提示
    const result = await processor.process({
        messages: [
            ...MessageV2.toModelMessages(msgs, model),
            ...(isLastStep ? [{
                role: "assistant",
                content: MAX_STEPS,   // 告知 AI 这是最后一步
            }] : []),
        ],
        tools,
        model,
    })
}
MAX_STEPS 机制：当 step 达到 maxSteps 时，不会强制终止循环，而是注入一条 assistant 消息（MAX_STEPS prompt），礼貌地告知 AI 不要再调用工具。AI 通常会在这一步给出最终总结。如果 AI 忽略提示继续调用工具，循环会继续（但不会再注入 MAX_STEPS）。

八、insertReminders — Plan/Build 上下文注入

insertReminders 是 loop 中的关键钩子，它根据当前 Agent 和历史 Agent 动态注入 system prompt：

三种注入场景

场景条件注入内容
进入 Plan 模式当前 agent=plan，上一条 assistant 非 plan完整的 Plan 工作流指引（5 个阶段）
Plan→Build 切换当前 agent=build，存在 plan assistant 消息BUILD_SWITCH + plan 文件内容
用户中途发消息step > 1 且有未处理的 user 消息<system-reminder> 包装用户消息
// 用户中途发送消息的处理
if (step > 1 && lastFinished) {
    for (const msg of msgs) {
        if (msg.info.role !== "user" || msg.info.id <= lastFinished.id) continue
        for (const part of msg.parts) {
            if (part.type !== "text" || part.ignored || part.synthetic) continue
            part.text = [
                "<system-reminder>",
                "The user sent the following message:",
                part.text,
                "",
                "Please address this message and continue with your tasks.",
                "</system-reminder>",
            ].join("\n")
        }
    }
}
为什么需要 system-reminder 包装？当 AI 正在执行多步任务时，用户可能中途插入新消息。如果不加标记，AI 可能将新消息误认为是之前对话的一部分。system-reminder 标签明确告知 AI"这是用户新发的消息"。

九、resolveTools — 工具解析全流程

resolveTools() 负责构建当前步骤的完整工具集：

1
内置工具注册 — ToolRegistry.tools(model, agent) 根据 Agent 权限过滤
2
MCP 工具注入 — MCP.tools() 添加所有已连接的 MCP 服务工具
3
Plugin Hook — 每次工具执行前/后触发 tool.execute.before/after
4
Schema 转换 — ProviderTransform.schema() 适配不同模型的 schema 差异
5
结构化输出 — 如果用户请求 JSON schema，注入 StructuredOutput 工具
// resolveTools 中的工具包装
tools[item.id] = tool({
    id: item.id,
    description: item.description,
    inputSchema: jsonSchema(schema),
    async execute(args, options) {
        const ctx = context(args, options)
        // Plugin hook: before
        await Plugin.trigger("tool.execute.before", { tool: item.id, ... }, { args })
        // 权限检查
        await ctx.ask({ permission: item.id, ... })
        // 实际执行
        const result = await item.execute(args, ctx)
        // Plugin hook: after
        await Plugin.trigger("tool.execute.after", { tool: item.id, ... }, output)
        return output
    },
})

十、SessionProcessor — 流式处理核心

SessionProcessor 是 ReAct 循环中处理 LLM 流输出的核心组件。它将 AI SDK 的流式事件转换为 OpenCode 的 Part 更新：

// session/processor.ts
export function create(input: {
    assistantMessage: MessageV2.Assistant
    sessionID: string
    model: Provider.Model
    abort: AbortSignal
}) {
    let snapshot: string | undefined
    let blocked = false
    let needsCompaction = false

    return {
        async process(streamInput) {
            while (true) {
                const stream = await LLM.stream(streamInput)
                for await (const value of stream.fullStream) {
                    switch (value.type) {
                        case "start-step":
                            snapshot = await Snapshot.track()   // 步骤开始时记录快照
                            await Session.updatePart({ type: "step-start", snapshot })
                            break
                        case "tool-call":
                            // 更新工具状态为 running
                            // 检测 doom loop（连续3次相同工具调用）
                            break
                        case "tool-result":
                            // 更新工具状态为 completed
                            break
                        case "tool-error":
                            // 如果是 PermissionDenied 或 QuestionRejected
                            if (error instanceof Question.RejectedError)
                                blocked = shouldBreak
                            break
                        case "finish-step":
                            // 计算 token 使用量和成本
                            // 记录步骤结束快照
                            // 检查是否需要 compaction
                            break
                        case "text-delta":
                            // 增量更新文本内容
                            break
                        case "reasoning-delta":
                            // 增量更新推理内容
                            break
                    }
                }
                // 返回控制信号
                if (needsCompaction) return "compact"
                if (blocked) return "stop"
                if (error) return "stop"
                return "continue"
            }
        }
    }
}

三种返回信号

返回值含义loop 行为
"continue"正常完成一步，AI 需要继续调用工具继续 while 循环
"stop"用户拒绝权限/问题，或发生错误退出循环
"compact"上下文即将溢出，需要压缩触发 SessionCompaction.create()，然后 continue

十一、Snapshot 追踪 — 每步 Diff 记录

每个 step 开始时，Processor 会记录一个 git tree hash 快照；step 结束时对比差异：

case "start-step":
    snapshot = await Snapshot.track()     // git add . && git write-tree → hash
    break

case "finish-step":
    if (snapshot) {
        const patch = await Snapshot.patch(snapshot)  // git diff hash --name-only
        if (patch.files.length) {
            await Session.updatePart({
                type: "patch",
                hash: patch.hash,
                files: patch.files,        // 本步骤修改的文件列表
            })
        }
        snapshot = undefined
    }
    break
Patch Part 的作用：记录每一步修改了哪些文件，前端据此展示实时 diff 视图。用户可以在任何一步"回退"（revert），将整个工作区恢复到该步骤开始前的状态。

十二、Compaction 触发与上下文溢出处理

在 finish-step 处理中，Processor 检查是否需要 Compaction：

case "finish-step":
    // 检查 token 使用量是否超过模型上下文的阈值
    if (!input.assistantMessage.summary &&
        await SessionCompaction.isOverflow({ tokens: usage.tokens, model: input.model })
    ) {
        needsCompaction = true
    }
    break

// 流处理结束后
if (needsCompaction) return "compact"

在 loop 中处理 compact 返回值：

// loop() 中
if (result === "compact") {
    // 创建 compaction 任务
    await SessionCompaction.create({
        sessionID,
        agent: lastUser.agent,
        model: lastUser.model,
        auto: true,
        overflow: !processor.message.finish,  // 是否因为溢出中断
    })
}
continue  // 继续循环，下次迭代会处理 compaction 任务

上下文溢出的两条路径

A
正常溢出：AI 完成了当前步骤（finish reason = stop），但 token 数超过阈值 → 自动压缩后继续
B
异常溢出：AI SDK 抛出 ContextOverflowError → 步骤中断，标记 overflow=true → 紧急压缩

十三、BusyError 与并发控制

每个 Session 同一时间只能有一个活跃的 ReAct 循环。并发保护通过 AbortController 状态字典实现：

// session/prompt.ts — state 字典
const state = Instance.state(() => {
    const data: Record<string, {
        abort: AbortController
        callbacks: { resolve, reject }[]
    }> = {}
    return data
})

// 启动循环时注册
function start(sessionID: string) {
    const s = state()
    if (s[sessionID]) return       // 已存在 → 返回 undefined → BusyError
    const controller = new AbortController()
    s[sessionID] = { abort: controller, callbacks: [] }
    return controller.signal
}

// 检查是否忙碌
export function assertNotBusy(sessionID: string) {
    const match = state()[sessionID]
    if (match) throw new Session.BusyError(sessionID)
}

// 第二个请求进入时的处理
if (!abort) {
    return new Promise((resolve, reject) => {
        // 加入等待队列
        state()[sessionID].callbacks.push({ resolve, reject })
    })
}
排队机制：如果第二个 prompt 请求到达时 Session 正在忙碌，它不会被拒绝，而是加入 callbacks 队列。当第一个循环完成后，队列中的回调会被 resolve，触发新的循环。

十四、标题生成 — ensureTitle 机制

在循环的第一步（step === 1），会异步触发标题生成：

// loop() 中
if (step === 1)
    ensureTitle({
        session,
        modelID: lastUser.model.modelID,
        providerID: lastUser.model.providerID,
        history: msgs,
    })

// ensureTitle 使用 title Agent
// title Agent 是 hidden Agent，prompt 为 PROMPT_TITLE
// 它的权限是 deny *（不使用任何工具），只做文本生成
异步非阻塞：标题生成是异步的，不会阻塞主循环。它使用独立的 title Agent（hidden、无工具权限），调用 generateText 而非 streamText，确保快速完成。

十五、完整生命周期总结

Session 生命周期核心规律

规律体现
