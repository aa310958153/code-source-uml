一、为什么需要快照系统？

AI 在 ReAct 循环中每一步都可能修改代码文件。如果没有快照机制，用户面临三个问题：

问题快照的解决方案
AI 改错了代码，如何回退？restore() 恢复到任意步骤的 git tree hash
这一步改了什么文件？patch() 对比两个 hash 之间的变更文件列表
具体改了什么内容？diffFull() 获取每个文件的 additions/deletions/before/after

二、私有 Git 仓库架构

OpenCode 的快照系统使用一个独立的私有 Git 仓库（不影响项目的 .git），存储在全局数据目录中：

// snapshot/index.ts
function gitdir() {
    const project = Instance.project
    return path.join(Global.Path.data, "snapshot", project.id)
    // 例如: ~/.local/share/opencode/snapshot/abc123/
}

// 初始化私有 Git 仓库
export async function track() {
    const git = gitdir()
    if (await fs.mkdir(git, { recursive: true })) {
        // 首次使用：初始化 Git 仓库
        await $`git init`.env({ GIT_DIR: git, GIT_WORK_TREE: Instance.worktree })
        // Windows 兼容配置
        await $`git --git-dir ${git} config core.autocrlf false`
        await $`git --git-dir ${git} config core.longpaths true`
        await $`git --git-dir ${git} config core.symlinks true`
        await $`git --git-dir ${git} config core.fsmonitor false`
    }
    await add(git)  // git add .
    const hash = await $`git --git-dir ${git} --work-tree ${Instance.worktree} write-tree`.text()
    return hash.trim()  // 返回 tree hash（不是 commit hash）
}
为什么用 tree hash 而非 commit hash？git write-tree 返回 tree hash，它只记录文件树状态，不需要 commit message。这比 commit 更轻量，速度更快，且足够用于 diff 和 restore。
与项目 Git 完全隔离：快照 Git 使用 GIT_DIR 环境变量指定独立的 git 目录，work-tree 指向项目目录。这意味着快照操作不会出现在项目的 git log 中。

三、track() — 每步快照记录

// processor.ts 中每个 step 的开始和结束
case "start-step":
    snapshot = await Snapshot.track()     // 记录当前文件状态
    await Session.updatePart({ type: "step-start", snapshot })
    break

case "finish-step":
    // ... 其他逻辑 ...
    await Session.updatePart({
        type: "step-finish",
        snapshot: await Snapshot.track(),  // 记录修改后的状态
    })
    break

每步记录两个 hash：step-start（修改前）和 step-finish（修改后）。两者的 diff 就是这一步的文件变更。

四、patch() — 变更文件检测

export async function patch(hash: string): Promise<Patch> {
    const git = gitdir()
    await add(git)
    const result = await $`git --git-dir ${git} --work-tree ${Instance.worktree}
        diff --no-ext-diff --name-only ${hash} -- .`
        .text()

    return {
        hash,
        files: result.trim().split("\n")
            .map(x => path.join(Instance.worktree, x).replaceAll("\\", "/"))
    }
}

patch() 返回从指定 hash 到当前工作区之间所有变更的文件路径列表。这些文件路径被存储为 patch Part，前端据此展示"本步骤修改了 N 个文件"。

五、diff() 与 diffFull() — 详细差异计算

// diff() — 返回 unified diff 文本
export async function diff(hash: string) {
    return await $`git --git-dir ${git} --work-tree ${Instance.worktree}
        diff --no-ext-diff ${hash} -- .`.text()
}

// diffFull() — 返回结构化的文件级差异
export async function diffFull(from: string, to: string): Promise<FileDiff[]> {
    // 1. 获取文件状态（added/deleted/modified）
    const statuses = await $`git diff --name-status --no-renames ${from} ${to}`.text()

    // 2. 获取行数统计（additions/deletions）
    for await (const line of $`git diff --numstat ${from} ${to}`.lines()) {
        const [additions, deletions, file] = line.split("\t")

        // 3. 获取 before/after 文件内容
        const before = await $`git show ${from}:${file}`.text()
        const after = await $`git show ${to}:${file}`.text()

        result.push({ file, before, after, additions, deletions, status })
    }
}

FileDiff 数据结构

export const FileDiff = z.object({
    file: z.string(),          // 文件路径
    before: z.string(),        // 修改前内容
    after: z.string(),         // 修改后内容
    additions: z.number(),     // 新增行数
    deletions: z.number(),     // 删除行数
    status: z.enum(["added", "deleted", "modified"]).optional(),
})

六、restore() 与 revert() — 快照回退

// restore() — 恢复到指定 tree hash 的完整状态
export async function restore(snapshot: string) {
    await $`git --git-dir ${git} --work-tree ${Instance.worktree}
        read-tree ${snapshot} &&
        git --git-dir ${git} --work-tree ${Instance.worktree}
        checkout-index -a -f`
}

// revert() — 精细回退：按文件恢复到各自的历史 hash
export async function revert(patches: Patch[]) {
    const files = new Set<string>()
    for (const item of patches) {
        for (const file of item.files) {
            if (files.has(file)) continue
            const result = await $`git checkout ${item.hash} -- ${file}`
            if (result.exitCode !== 0) {
                // 文件在快照中不存在 → 删除
                const checkTree = await $`git ls-tree ${item.hash} -- ${file}`
                if (!checkTree.text().trim())
                    await fs.unlink(file)
            }
            files.add(file)
        }
    }
}
restore vs revert：restore 是将整个工作区恢复到一个快照点；revert 是精细操作 — 对每个文件分别恢复到其最后一次被修改前的快照版本。revert 更常用，因为它允许用户选择性回退。

七、Patch 模块 — apply-patch 格式解析

除了 Git 快照，OpenCode 还实现了一个 apply-patch 格式解析器（patch/index.ts），用于处理 AI 生成的结构化代码修改：

// 支持的 patch 操作
export type Hunk =
    | { type: "add"; path: string; contents: string }
    | { type: "delete"; path: string }
    | { type: "update"; path: string; move_path?: string; chunks: UpdateFileChunk[] }

// Patch 格式示例
*** Begin Patch
*** Add File: src/new-feature.ts
+export function newFeature() {
+    return "hello"
+}
*** Update File: src/index.ts
@@ main function
 export function main() {
-    console.log("old")
+    console.log("new")
 }
*** Delete File: src/deprecated.ts
*** End Patch

apply-patch 的安全验证

Patch 模块实现了两级验证：先验证 patch 语法是否正确（Body），再验证应用 patch 后的结果是否符合预期（CorrectnessError）。这确保了 AI 生成的代码修改不会破坏文件。

八、与 Processor 的集成

快照系统与 SessionProcessor 紧密集成，每步 ReAct 循环自动记录变更：

九、自动清理与 GC

// 每小时自动清理 7 天前的快照
export function init() {
    Scheduler.register({
        id: "snapshot.cleanup",
        interval: hour,
        run: cleanup,
        scope: "instance",
    })
}

export async function cleanup() {
    await $`git --git-dir ${git} --work-tree ${Instance.worktree} gc --prune=7.days`
}

Git GC 会压缩和清理不可达的对象，防止快照仓库无限增长。

十、完整生命周期总结

规律体现
私有 Git独立 GIT_DIR，不污染项目 Git 历史
Tree Hash使用 write-tree 而非 commit，轻量高效
每步双记start-step 和 finish-step 各记一次 hash，精确计算步骤变更
精细回退revert 按文件级别回退，restore 按整体回退
结构化 DiffdiffFull 返回 before/after 内容 + additions/deletions 统计
apply-patch独立于 Git 的结构化代码修改格式，带安全验证
自动 GC每小时清理 7 天前的不可达对象，防止仓库膨胀
