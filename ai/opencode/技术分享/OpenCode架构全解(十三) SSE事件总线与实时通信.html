一、事件驱动架构的必要性

OpenCode 是典型的 C/S 架构：后端（Node.js/Bun 进程）执行 AI 推理和工具调用，前端（Web/TUI/Desktop）展示结果。两端之间需要一个高效、低延迟的实时通信机制。

为什么选择 SSE 而非 WebSocket？

维度SSEWebSocket
方向单向（服务端→客户端）双向
复杂度基于 HTTP，天然兼容代理/防火墙独立协议，需要额外配置
断线重连浏览器原生支持 Last-Event-ID需手动实现
适合场景事件推送（OpenCode 的核心需求）双向实时交互

OpenCode 的客户端→服务端通信走 REST API（POST /session/:id/chat），服务端→客户端走 SSE。职责清晰分离。

二、四层事件传播架构

三、Bus — Instance 级发布/订阅

// packages/opencode/src/bus/index.ts
export namespace Bus {
    type Subscription = (event: any) => void

    const state = Instance.state(() => {
        const subscriptions = new Map<any, Subscription[]>()
        return { subscriptions }
    })

    export async function publish(def, properties) {
        const payload = { type: def.type, properties }
        // 通知本 type 的订阅者
        for (const key of [def.type, "*"]) {
            const match = state().subscriptions.get(key)
            for (const sub of match ?? []) pending.push(sub(payload))
        }
        // 同时发送到全局 EventEmitter
        GlobalBus.emit("event", { directory: Instance.directory, payload })
        return Promise.all(pending)
    }

    export function subscribe(def, callback) {
        return raw(def.type, callback)
    }

    export function subscribeAll(callback) {
        return raw("*", callback)   // 通配符订阅所有事件
    }

    export function once(def, callback) {
        const unsub = subscribe(def, (event) => {
            if (callback(event)) unsub()
        })
    }
}
两种订阅模式：subscribe(Event.Created, cb) 精确订阅特定事件类型；subscribeAll(cb) 通配符订阅所有事件。Plugin 系统使用 subscribeAll 监听全局事件。

四、BusEvent.define — 类型安全的事件定义

// packages/opencode/src/bus/bus-event.ts
export namespace BusEvent {
    const registry = new Map<string, Definition>()

    export function define(type, properties) {
        const result = { type, properties }
        registry.set(type, result)     // 注册到全局 registry
        return result
    }

    // 基于 registry 自动生成 Zod discriminatedUnion 类型
    export function payloads() {
        return z.discriminatedUnion("type",
            registry.entries().map(([type, def]) =>
                z.object({
                    type: z.literal(type),
                    properties: def.properties,
                })
            ).toArray()
        )
    }
}

定义示例

// 各模块定义自己的事件
Session.Event.Created = BusEvent.define("session.created", z.object({ info: Info }))
Session.Event.Updated = BusEvent.define("session.updated", z.object({ info: Info }))
MessageV2.Event.Updated = BusEvent.define("message.updated", z.object({ info: MessageV2.Info }))
MessageV2.Event.PartUpdated = BusEvent.define("part.updated", z.object({ part: MessageV2.Part }))
Question.Event.Asked = BusEvent.define("question.asked", Question.Request)
PermissionNext.Event.Asked = BusEvent.define("permission.asked", ...)
MCP.ToolsChanged = BusEvent.define("mcp.tools.changed", z.object({ server: z.string() }))
Event.Connected = BusEvent.define("server.connected", z.object({}))

payloads() 方法将所有注册的事件自动生成 Zod discriminatedUnion，用于 OpenAPI schema 导出和 SSE 端点类型定义。

五、GlobalBus — 跨 Instance 事件汇聚

// packages/opencode/src/bus/global.ts — 极简实现
import { EventEmitter } from "events"

export const GlobalBus = new EventEmitter<{
    event: [{ directory?: string, payload: any }]
}>()

GlobalBus 是一个 Node.js EventEmitter，它的唯一职责是跨项目汇聚事件。一个 OpenCode 服务可能同时管理多个项目（Instance），每个项目有自己的 Bus。GlobalBus 将所有 Instance 的事件统一到一个流中。

为什么需要两层 Bus？

一个 OpenCode 进程可以同时打开多个项目目录（workspace）。每个目录是一个独立的 Instance，有自己的 Session、Agent、Tool 状态。但前端 SSE 连接只有一个（GET /global/event），需要接收所有项目的事件。

GlobalBus 就是这个汇聚点：每个 Instance 的 Bus.publish 同时向 GlobalBus 发事件，SSE 端点只需监听 GlobalBus。

六、SSE 服务端 — Hono streamSSE

// packages/opencode/src/server/routes/global.ts
.get("/event", async (c) => {
    c.header("X-Accel-Buffering", "no")      // 禁止 Nginx 缓冲
    c.header("X-Content-Type-Options", "nosniff")
    return streamSSE(c, async (stream) => {
        // 连接成功时发送初始事件
        stream.writeSSE({
            data: JSON.stringify({
                payload: { type: "server.connected", properties: {} },
            }),
        })

        // 监听 GlobalBus
        async function handler(event: any) {
            await stream.writeSSE({ data: JSON.stringify(event) })
        }
        GlobalBus.on("event", handler)

        // 10 秒心跳，防止代理超时
        const heartbeat = setInterval(() => {
            stream.writeSSE({
                data: JSON.stringify({
                    payload: { type: "server.heartbeat", properties: {} },
                }),
            })
        }, 10_000)

        await new Promise((resolve) => {
            stream.onAbort(() => {
                clearInterval(heartbeat)
                GlobalBus.off("event", handler)
                resolve()
            })
        })
    })
})
关键细节：
  • X-Accel-Buffering: no — 防止 Nginx 等反向代理缓冲 SSE 数据，确保事件实时到达
  • heartbeat — 每 10 秒发送心跳，防止空闲连接被代理/防火墙断开
  • stream.onAbort — 客户端断开时自动清理监听器，防止内存泄漏

七、SSE 客户端 — parseSSE 解析器

// packages/opencode/src/control-plane/sse.ts
export async function parseSSE(body, signal, onEvent) {
    const reader = body.getReader()
    const decoder = new TextDecoder()
    let buf = ""
    let last = ""
    let retry = 1000

    while (!signal.aborted) {
        const chunk = await reader.read()
        if (chunk.done) break
        buf += decoder.decode(chunk.value, { stream: true })

        // 统一换行符
        buf = buf.replace(/\r\n/g, "\n").replace(/\r/g, "\n")

        // SSE 协议：双换行分隔事件
        const chunks = buf.split("\n\n")
        buf = chunks.pop() ?? ""   // 最后一块可能不完整，保留

        chunks.forEach((chunk) => {
            const data: string[] = []
            chunk.split("\n").forEach((line) => {
                if (line.startsWith("data:"))
                    data.push(line.replace(/^data:\s*/, ""))
                if (line.startsWith("id:"))
                    last = line.replace(/^id:\s*/, "")
                if (line.startsWith("retry:"))
                    retry = parseInt(line.replace(/^retry:\s*/, ""))
            })
            if (data.length) {
                const raw = data.join("\n")
                try { onEvent(JSON.parse(raw)) }
                catch { onEvent({ type: "sse.message", properties: { data: raw, id: last } }) }
            }
        })
    }
}

九、前端 EventReducer — 状态同步

前端通过 EventReducer 将 SSE 事件映射为 SolidJS Store 的状态更新：

事件处理模式

// packages/app/src/context/global-sync/event-reducer.ts
// 简化示例
function reduce(state, event) {
    switch (event.payload.type) {
        case "session.created":
            state.sessions[event.payload.properties.info.id] = event.payload.properties.info
            break
        case "message.updated":
            // 更新消息列表
            break
        case "part.updated":
            // 更新消息 parts
            const part = event.payload.properties.part
            state.parts[part.messageID] = [...existing, part]
            break
        case "part.delta":
            // 增量更新文本（流式渲染）
            const { partID, field, delta } = event.payload.properties
            state.parts[msgID].find(p => p.id === partID)[field] += delta
            break
        case "question.asked":
            // 展示问题 UI
            state.pendingQuestions[requestID] = event.payload.properties
            break
    }
}

part.delta 是最高频的事件 — AI 每生成一个 token 就触发一次，前端据此实现逐字打字效果。

十、心跳与断线重连

服务端心跳

每 10 秒发送 server.heartbeat 事件，防止空闲连接被 Nginx/CDN/防火墙断开。这是 SSE 在生产环境中的标准实践。

客户端重连

// control-plane/workspace.ts 中的 SSE 重连
await parseSSE(res.body, stop, (event) => {
    // 处理事件...
})
// SSE 连接断开后
// Wait 250ms and retry if SSE connection fails
await new Promise(r => setTimeout(r, 250))
// 重新建立连接...

客户端在连接断开后等待 250ms 重试，采用指数退避策略避免重连风暴。由于 SSE 协议的特性，重连后会从最后接收的事件 ID 继续，不会丢失事件。

十一、Database.effect — 事务后发布

一个精巧的设计：事件发布必须等待数据库事务提交后才执行，防止前端收到事件后查询到旧数据。

// Database.use 中的 effect 机制
Database.use((db) => {
    // 1. 执行数据库写入
    db.insert(SessionTable).values(toRow(result)).run()

    // 2. 注册事务后回调（不会立即执行）
    Database.effect(() =>
        Bus.publish(Event.Created, { info: result })
    )
    // 3. 事务提交后，effect 回调才执行 → 事件才发出
})
为什么这很重要？如果事件在事务提交前发出，前端收到事件后立即请求 Session 详情，可能拿到的是事务前的旧数据（或找不到）。Database.effect 确保"事件到达时，数据已经持久化"。
