一、Sensors 总览：Agent 的"神经系统"

用人体隐喻理解 Sensors：

如果 LLM 是 Agent 的"大脑"，工具是"眼睛"和"双手"，那 Sensors 就是神经系统——
• 大脑说"伸手拿东西" → 神经检查：手能伸那么远吗？拿的东西安全吗？
• 眼睛看到东西 → 神经传递：信息量太大？截断一点再传给大脑
• 手在重复同一个动作 → 神经警告：你在死循环了，停下来想想
• 手要碰危险物品 → 神经拦截：等等，先问问主人
Sensor核心文件介入时机触发条件失败动作

二、Sensor 1：Zod Schema — 输入校验

LLM 不是程序员，它生成的工具调用参数可能不合法（缺少字段、类型错误、路径不存在）。Zod Schema 是第一道防线。

2.1 实现机制

// tool/tool.ts:58-69 — 工具定义的 execute 被包装

toolInfo.execute = async (args, ctx) => {
    try {
        toolInfo.parameters.parse(args)    // ← Zod 校验！
    } catch (error) {
        if (error instanceof z.ZodError && toolInfo.formatValidationError) {
            // 工具可以提供自定义错误格式
            throw new Error(toolInfo.formatValidationError(error), {cause: error})
        }
        throw new Error(
            `The ${id} tool was called with invalid arguments: ${error}.\n
             Please rewrite the input so it satisfies the expected schema.`,
            {cause: error},
        )
    }
    const result = await execute(args, ctx)
    // ...truncation 处理...
}
设计精妙点：
• 错误消息是写给 LLM 看的："Please rewrite the input so it satisfies the expected schema"
• LLM 收到错误后，会自然地修正参数重新调用 → 自动纠错循环
• 每个工具可以自定义 formatValidationError()，提供更精准的修正指导
• 所有工具通过 Tool.define() 统一注册，校验逻辑集中在一个地方

三、Sensor 2：Output Truncation — 输出截断

工具输出可能非常大（读取一个 10 万行的日志文件），直接发给 LLM 会浪费大量 token。Truncation 是输出侧的安全阀。

3.1 硬限制

// truncation.ts:12-13

export const MAX_LINES = 2000      // 最多 2000 行
export const MAX_BYTES = 50 * 1024 // 最多 50KB

3.2 截断 + 保存 + 建议

// truncation.ts:56-111

export async function output(text, options = {}, agent?) {
    const lines = text.split("\n")
    if (lines.length <= maxLines && totalBytes <= maxBytes)
        return { content: text, truncated: false }  // 不超限，直接返回

    // 1. 截取前/后 maxLines 行（或 maxBytes）
    // 2. 将完整输出保存到文件系统
    const filepath = path.join(DIR, id)
    await Filesystem.write(filepath, text)

    // 3. 根据 Agent 能力给出不同建议
    const hint = hasTaskTool(agent)
        ? "Use the Task tool to have explore agent process this file"
        : "Use Grep to search the full content or Read with offset/limit"

    return { content: preview + "...N lines truncated..." + hint, truncated: true }
}
与 Task 工具的策略联动：
• 如果 Agent 有 Task 工具 → 建议"派子 Agent 处理这个文件"（上下文隔离）
• 如果 Agent 没有 Task 工具 → 建议"用 Grep 搜索或 Read 分段查看"
• 完整输出保存为文件 → Agent 可以后续用 Read 工具分段查看
• 7 天自动清理：Scheduler 每小时扫描，删除超过 7 天的截断文件

四、Sensor 3：Doom Loop Detection — 死循环检测

LLM 有时会陷入"死亡循环"——反复用完全相同的参数调用同一个工具（比如一直 Read 同一个不存在的文件）。Doom Loop 检测是 Agent 的"精神健康监护"。

4.1 检测算法

// processor.ts:20, 150-176

const DOOM_LOOP_THRESHOLD = 3

// 在 tool-call 事件处理中：
const parts = await MessageV2.parts(input.assistantMessage.id)
const lastThree = parts.slice(-DOOM_LOOP_THRESHOLD)

if (
    lastThree.length === DOOM_LOOP_THRESHOLD &&
    lastThree.every((p) =>
        p.type === "tool" &&                          // 都是工具调用
        p.tool === value.toolName &&                   // 同一个工具
        p.state.status !== "pending" &&                // 不是 pending 状态
        JSON.stringify(p.state.input)
            === JSON.stringify(value.input)            // ← 完全相同的参数！
    )
) {
    // 触发权限询问！
    await PermissionNext.ask({
        permission: "doom_loop",
        patterns: [value.toolName],
        sessionID: input.assistantMessage.sessionID,
        metadata: { tool: value.toolName, input: value.input },
        always: [value.toolName],
        ruleset: agent.permission,
    })
}
设计精妙点：
• 使用 JSON.stringify 深度比较参数——即使参数是复杂对象也能精确匹配
• 检测的是同一轮 Assistant 消息中的工具调用，不会跨轮次误判
• 触发后走 Permission 系统——用户可以选"允许继续"或"拒绝并给出指导"
• 阈值 3：1 次可能是偶然，2 次可能是在尝试，3 次大概率是死循环

五、Sensor 4：Permission System — 权限守卫

Permission 系统是 Agent 行为的"法律体系"——定义了 Agent 能做什么、不能做什么、以及需要确认什么。

5.1 三级权限模型

Action含义行为来源
allow允许直接执行AGENTS.md / 用户授权 "always"
deny拒绝抛 DeniedError，停止执行AGENTS.md 规则
ask询问阻塞等待用户确认默认（无匹配规则时）

5.2 权限评估流程

// permission/next.ts:135-165

export const ask = fn(..., async (input) => {
    for (const pattern of request.patterns ?? []) {
        const rule = evaluate(request.permission, pattern, ruleset, approved)

        if (rule.action === "deny")
            throw new DeniedError(...)    // ← 硬拒绝

        if (rule.action === "ask") {
            // 创建 Promise，阻塞等待用户回复
            return new Promise<void>((resolve, reject) => {
                s.pending[id] = { info, resolve, reject }
                Bus.publish(Event.Asked, info)  // → SSE 推送给用户
            })
        }

        if (rule.action === "allow") continue  // 放行
    }
})

// evaluate: 最后匹配的规则生效（last match wins）
export function evaluate(permission, pattern, ...rulesets) {
    const merged = merge(...rulesets)
    const match = merged.findLast(
        (rule) => Wildcard.match(permission, rule.permission)
               && Wildcard.match(pattern, rule.pattern),
    )
    return match ?? { action: "ask", permission, pattern: "*" }  // 默认 ask
}

5.3 用户回复处理

1
用户选择 "once"（本次允许）
resolve Promise → 工具执行继续。下次同样的操作还是会询问。
2
用户选择 "always"（永久允许）
将规则加入 approved 列表 → 同 Session 内后续相同操作自动允许。还会自动 resolve 其他匹配的 pending 请求。
3
用户选择 "reject"（拒绝）
两种拒绝方式：
• RejectedError：单纯拒绝，Agent 停止当前操作
• CorrectedError：带消息拒绝（"不要用rm，用trash"），Agent 收到指导后修正行为
同时 reject 还会批量取消同 Session 的所有 pending 权限请求！
权限层级叠加：
• Agent 定义（agent.ts）：每个 Agent 有默认权限集（explore Agent 只允许 grep/read）
• Session 级别：可以在 Session 上设置额外权限
• 用户授权：运行时用户 approve 的规则
• 三层合并后，最后匹配的规则优先（last match wins）

六、Sensor 5：Bash AST Security — 命令安全解析

Bash 是最危险的工具——它可以执行任意命令。OpenCode 使用 tree-sitter 对命令进行 AST 级别解析，提取命令名、参数、路径，然后进行安全检查。

6.1 AST 解析流程

// bash.ts:84-142

// 1. 用 tree-sitter 解析命令为 AST
const tree = await parser().then(p => p.parse(params.command))

// 2. 遍历所有 command 节点
for (const node of tree.rootNode.descendantsOfType("command")) {
    const command = []
    for (let i = 0; i < node.childCount; i++) {
        const child = node.child(i)
        // 只提取 command_name, word, string 类型节点
        command.push(child.text)
    }

    // 3. 对文件操作命令（cd/rm/cp/mv/mkdir...）检查路径
    if (["cd", "rm", "cp", "mv", "mkdir", "touch", "chmod", "chown", "cat"]
        .includes(command[0])) {
        for (const arg of command.slice(1)) {
            const resolved = await $`realpath ${arg}`.text()
            if (resolved && !Instance.containsPath(normalized)) {
                directories.add(dir)  // 外部目录！需要权限
            }
        }
    }

    // 4. 非 cd 命令 → 请求 bash 权限
    if (command.length && command[0] !== "cd") {
        patterns.add(commandText)
        always.add(BashArity.prefix(command).join(" ") + " *")
    }
}

6.2 BashArity：命令粒度识别

BashArity 是一个精巧的字典表，用于从完整命令中提取"人类可理解的命令"：

完整命令提取的命令Arity用途
rm -rf /tmp/foorm1权限检查用
git checkout maingit checkout2允许所有 git checkout
npm run devnpm run dev3只允许这个具体脚本
docker compose up -ddocker compose3允许所有 docker compose
Bash 安全的三层防护：
• 第 1 层：AST 解析 → 提取命令名和参数，不是简单的字符串匹配
• 第 2 层：路径检查 → realpath 解析后，检查是否在项目目录外
• 第 3 层：权限请求 → 外部目录用 external_directory，命令用 bash 权限
额外的安全机制：默认 2 分钟超时 → 自动 kill 进程树

七、Sensor 6：Retry with Backoff — 自动重试

LLM API 不是 100% 稳定的——限流、过载、网络波动时有发生。Retry 机制让 Agent 能自动恢复，而不是遇到一个 429 就彻底停摆。

7.1 重试策略

// retry.ts:6-9, 28-58

const RETRY_INITIAL_DELAY  = 2000        // 初始延迟 2 秒
const RETRY_BACKOFF_FACTOR = 2           // 指数退避因子
const RETRY_MAX_DELAY      = 30_000      // 最大延迟 30 秒

export function delay(attempt, error?) {
    // 1. 优先使用服务器返回的 Retry-After 头
    if (headers["retry-after-ms"])
        return parseFloat(headers["retry-after-ms"])
    if (headers["retry-after"])
        return parseFloat(headers["retry-after"]) * 1000

    // 2. 否则使用指数退避：2s → 4s → 8s → 16s → 30s（封顶）
    return Math.min(
        RETRY_INITIAL_DELAY * Math.pow(2, attempt - 1),
        RETRY_MAX_DELAY,
    )
}

7.2 可重试 vs 不可重试

// retry.ts:61-99

export function retryable(error) {
    if (ContextOverflowError.isInstance(error)) return undefined  // 不重试

    if (APIError.isInstance(error)) {
        if (!error.data.isRetryable) return undefined    // 不重试
        if (error.data.responseBody?.includes("FreeUsageLimitError"))
            return "Free usage exceeded"                 // 提示但不重试
        return error.data.message                        // 可重试
    }

    // 识别特定的可重试错误
    if (json.error?.type === "too_many_requests") return "Too Many Requests"
    if (code.includes("exhausted")) return "Provider is overloaded"
    if (code.includes("rate_limit")) return "Rate Limited"
    return undefined  // 未知错误不重试
}
重试机制在 Processor 中的位置：
在 processor.ts 的 catch 块中，先用 SessionRetry.retryable() 判断错误是否可重试。
如果可以 → 计算延迟 → sleep(delay, abort) → continue 回到 while(true) 循环顶部重试。
如果不可以 → 设置 error → 返回 "stop"。

八、Sensor 7：Snapshot & Revert — 变更追踪与回滚

Agent 写代码时可能犯错——把文件改坏了怎么办？Snapshot 机制在每一步开始时拍照，随时可以回滚。

8.1 实现原理

// snapshot/index.ts — 利用 Git 的内部机制

// 每个 step 开始时：
case "start-step":
    snapshot = await Snapshot.track()     // 创建一个 Git tree hash

// track() 做了什么：
export async function track() {
    await add(git)  // git add 所有文件
    const hash = await $`git write-tree`.text()  // 生成 tree hash
    return hash.trim()
}

// step 结束时：
case "finish-step":
    const patch = await Snapshot.patch(snapshot)  // 对比前后差异
    if (patch.files.length) {
        await Session.updatePart({ type: "patch", hash, files: patch.files })
    }

8.2 回滚能力

// snapshot/index.ts:116-133, 135-167

// 恢复到指定快照
export async function restore(snapshot) {
    await $`git read-tree ${snapshot} && git checkout-index -a -m`
}

// 按文件粒度回滚
export async function revert(patches) {
    for (const item of patches) {
        for (const file of item.files) {
            // 恢复每个文件到快照时的状态
            await $`git checkout ${item.hash} -- ${file}`
        }
    }
}
Snapshot 的设计优势：
• 利用 Git 基础设施：不自己管理文件差异，复用 git write-tree / git diff
• 独立 Git 目录：使用单独的 .git 目录，不污染项目的 Git 仓库
• 文件级粒度：可以回滚单个文件，不需要全量恢复
• 自动清理：每小时运行 git gc --prune=7.days，清理过期快照

十一、设计优势分析

🛡️

纵深防御

每个工具调用经过 4-5 层 Sensor 检查（Zod → Doom → Permission → Bash → Truncation），不是单点防护而是层层设防。

任何一层不通过都不会执行
🔄

自动纠错循环

Zod 校验失败时，错误消息引导 LLM 修正参数重试。Permission 拒绝时，用户可以给出指导性消息。Agent 能自我修复。

LLM 读到错误 → 修正 → 再试
🔍

AST 级精确

Bash 安全不是简单的黑名单字符串匹配，而是用 tree-sitter 解析 AST，精确提取命令名、参数、重定向路径。

rm -rf / | grep foo → 精确识别 rm 和路径
⚡

零性能开销

Zod 校验是内存操作（微秒级），Doom Loop 只检查最近 3 个 Part，Permission 评估是数组 findLast。几乎无感知。

不增加用户等待时间
🎯

对用户透明

大部分 Sensor 在后台默默工作。只有 Permission "ask" 和 Doom Loop 会打扰用户——且只在必要时。

"always" 授权后不再打扰
🔌

可配置可插拔

权限规则可配置在 AGENTS.md 或 Session 上；Truncation 限制可调；Bash 超时可设；Compaction 可关闭。灵活适配。

不同场景不同配置

Sensors 与其他 Harness 组件的关系

Harness 组件 人体隐喻 与 Sensors 的关系 核心文件
Guides（引导） 大脑的知识 Sensors 执行引导中定义的约束（AGENTS.md 中的 deny 规则） instruction.ts, agent.ts
Toolchain（工具链） 眼睛和手 Sensors 在工具调用前后包裹执行，是工具的"安全壳" tool.ts, bash.ts, edit.ts
Sensors（传感器） 神经系统 本文档的主题 — 检测异常、校验输入、截断输出 processor.ts, truncation.ts, permission/
Memory（记忆） 记忆系统 Sensors 的 Snapshot 提供回滚能力；isOverflow 触发 Compaction compaction.ts, snapshot/
Lifecycle（生命周期） 作息规律 Sensors 的 Retry 管理 API 故障恢复；Doom Loop 管理行为异常 retry.ts, processor.ts
总结：Sensors 是 Harness 设计范式中不可或缺的"神经系统"

在提示词工程时代，没有 Sensors——LLM 输出什么就是什么。
在上下文工程时代，Sensors 还很简陋——可能只有一个长度截断。
在 Harness 设计范式中，Sensors 是一套完整的、分层的、可配置的反馈系统——
确保 Agent 的每一次"看"和"做"都是安全的、受控的、可追溯的。
