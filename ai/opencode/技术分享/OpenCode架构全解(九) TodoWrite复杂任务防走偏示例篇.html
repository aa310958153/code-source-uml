场景设定与问题定义

用户输入

用户："帮我给这个 Express 项目添加完整的用户认证系统，包括 JWT 令牌、用户注册登录 API、密码加密、数据库模型、单元测试，以及更新 CI 配置"

为什么这个任务容易走偏？

走偏风险具体表现
范围蔓延模型可能开始优化无关代码、重构现有架构、修改不相关的配置
遗忘子任务在 8+ 个子任务中，修完 Bug 后可能忘记回到 CI 配置或测试
重复循环反复读同一个文件、反复修改同一处代码（Doom Loop）
步数爆炸无规划地执行导致循环次数超限，被强制终止
子任务串扰子 Agent 尝试修改父 Agent 的任务列表，造成状态混乱

TodoWrite 工具架构全景

三层架构：工具定义 → 数据模型 → 存储

// ─── ① 工具定义层 (src/tool/todo.ts) ───
export const TodoWriteTool = Tool.define("todowrite", {
    description: DESCRIPTION_WRITE,    // ← 从 todowrite.txt 加载（168行指令）
    parameters: z.object({
        todos: z.array(z.object(Todo.Info.shape)),
        //        ↑ 直接引用数据模型定义，保证类型一致
    }),
    async execute(params, ctx) {
        // 权限检查 — 所有工具统一走 PermissionNext.ask()
        await ctx.ask({
            permission: "todowrite",
            patterns: ["*"],
            always: ["*"],         // ← 每次都需检查，不自动放行
            metadata: {},
        })
        // 持久化到 SQLite
        await Todo.update({
            sessionID: ctx.sessionID,
            todos: params.todos,
        })
        return {
            title: `${params.todos.filter(x => x.status !== "completed").length} todos`,
            //        ↑ 标题动态计算未完成数，在 UI 上实时展示
            output: JSON.stringify(params.todos, null, 2),
            metadata: { todos: params.todos },
        }
    },
})

// ─── ② 数据模型层 (src/session/todo.ts) ───
export namespace Todo {
    export const Info = z.object({
        content: z.string().describe("Brief description of the task"),
        status:  z.string().describe("pending | in_progress | completed | cancelled"),
        priority: z.string().describe("high | medium | low"),
    })
    // 状态机：pending → in_progress → completed
    //                  → cancelled
}

// ─── ③ 存储层 (SQLite) ───
// TodoTable: session_id | content | status | priority | position
// 写入策略：先 DELETE 全部，再 INSERT 新列表（全量覆盖）
// 事件发布：Bus.publish(Event.Updated, ...) → UI 实时刷新
关键设计决策：Todo 列表采用"全量替换"而非增量更新。每次调用 TodoWrite 都会删除旧列表、写入新列表。这简化了状态管理，但也意味着模型必须在每次调用时提供完整列表。

权限隔离：子任务为什么不能操作 Todo？

// src/tool/task.ts — 创建子会话时注入权限规则
const session = await Session.create({
    parentID: ctx.sessionID,
    title: params.description + ` (@${agent.name} subagent)`,
    permission: [
        { permission: "todowrite", pattern: "*", action: "deny" },  // ← 禁止写 Todo
        { permission: "todoread", pattern: "*", action: "deny" },   // ← 禁止读 Todo
        // 子 Agent 无法修改父 Agent 的任务列表
    ],
})

// 同时在 tools 层面也禁用
const result = await SessionPrompt.prompt({
    sessionID: session.id,
    tools: {
        todowrite: false,   // ← 工具直接不注册
        todoread: false,
    },
    ...
})
双重隔离：子 Agent 既在 permission 层面被 deny，又在 tools 注册层面被禁用。这意味着即使子 Agent 尝试调用 TodoWrite，工具根本不存在于它的可用工具集中。

Phase 2：走偏风险 — system-reminder 如何纠偏

走偏风险

场景：用户在 Step 4 中途插入消息

用户（中途插入）："对了，顺便帮我看看 src/utils/format.ts 有个 lint 警告能不能修一下"

这是一个典型的走偏场景：用户在任务进行中追加了一个与主线无关的请求。如果不加处理，模型可能修完 lint 后忘记回到认证任务。

Step 5: 模型看到的用户消息被 system-reminder 包装

// prompt.ts 第 631-647 行 — step > 1 时的消息包装
if (step > 1 && lastFinished) {
    for (const msg of msgs) {
        if (msg.info.role !== "user" || msg.info.id <= lastFinished.id) continue
        for (const part of msg.parts) {
            if (part.type !== "text" || part.ignored || part.synthetic) continue
            if (!part.text.trim()) continue
            part.text = [
                "<system-reminder>",
                "The user sent the following message:",
                part.text,
                "",
                "Please address this message and continue with your tasks.",
                "</system-reminder>",
            ].join("\n")
        }
    }
}

[tool call]: Edit({ filePath: "src/utils/format.ts", ... })

    todos: [
        ...前 3 个 completed,
        { content: "实现注册 API",         status: "completed", priority: "high" },

        // ...
    ]

防走偏第二层 — system-reminder 包装：当 step > 1 时，用户新发的消息被包裹在 <system-reminder> 标签中，并附加"and continue with your tasks"指令。这明确告诉模型：处理完用户的新请求后，必须回到之前的任务。

Phase 3：Doom Loop 检测 — 同一工具重复调用拦截

Doom Loop 风险

场景：模型在实现登录 API 时陷入循环

模型修改 login 路由后发现类型错误，反复 Read 同一个文件试图理解问题：

processor.ts — Doom Loop 检测核心逻辑

// src/session/processor.ts
const DOOM_LOOP_THRESHOLD = 3

// 在 "tool-call" 事件处理中：
const parts = await MessageV2.parts(input.assistantMessage.id)
const lastThree = parts.slice(-DOOM_LOOP_THRESHOLD)

if (
    lastThree.length === DOOM_LOOP_THRESHOLD &&
    lastThree.every(p =>
        p.type === "tool" &&
        p.tool === value.toolName &&                    // ← 同一个工具
        p.state.status !== "pending" &&
        JSON.stringify(p.state.input) ===               // ← 相同参数
            JSON.stringify(value.input)
    )
) {
    // 触发 Doom Loop 保护 — 要求用户确认
    await PermissionNext.ask({
        permission: "doom_loop",
        patterns: [value.toolName],
        sessionID: input.assistantMessage.sessionID,
        metadata: {
            tool: value.toolName,
            input: value.input,
        },
    })
}

实际触发场景

// 模型连续 3 次调用 Read，参数完全相同：
tool-call #1: Read({ filePath: "src/routes/auth.ts" }) → 返回文件内容
tool-call #2: Read({ filePath: "src/routes/auth.ts" }) → 返回文件内容
tool-call #3: Read({ filePath: "src/routes/auth.ts" }) → ⚠️ Doom Loop!

// 系统弹出确认对话框：
"模型正在重复调用 Read 工具（参数相同），是否允许继续？"
  → [允许] → 模型继续（可能确实需要刷新文件内容）
  → [拒绝] → processor 返回 "stop"，循环中断

// 这防止了：
// 1. 无意义的 token 消耗（反复读同一个大文件）
// 2. 模型在死循环中耗尽 max-steps
// 3. 上下文被重复的工具结果淹没
防走偏第三层 — Doom Loop 检测：当检测到连续 3 次相同工具 + 相同参数的调用时，系统暂停执行并要求用户确认。这防止了模型在死循环中浪费资源和步数。

六层防走偏机制全景图

从用户输入到最终输出的完整防线

各层防线的触发时机

防线触发位置触发条件作用
TodoWrite 模型自主决策 复杂任务（3+ 步骤） 创建任务清单，锚定后续行为
system-reminder prompt.ts loop() step > 1 && 用户有新消息 包装消息 + 提醒"继续你的任务"
Doom Loop processor.ts 连续 3 次相同工具+参数 暂停执行，要求用户确认
权限隔离 task.ts 创建子 Agent 会话时 deny todowrite/todoread
max-steps prompt.ts loop() step >= agent.steps 禁止工具，强制输出总结
Compaction prompt.ts + processor.ts token 超限 或 finish-step 溢出 压缩历史，保留关键信息

长任务中 Todo 如何不被上下文淹没

核心问题：10 步任务，前 8 步上下文已经爆炸

假设一个 10 步任务，每步调用 2-3 个工具（Read/Edit/Bash），每步产生 5K-20K tokens 的工具输出。到第 8 步时：

Step累积工具调用估算 tokensTodoWrite 输出状态
1TodoWrite + Read + Glob~8K完整可见
2TodoWrite + Read + Edit~20K完整可见
3TodoWrite + Edit + Edit~35K完整可见
4TodoWrite + Read + Bash~55K完整可见
5TodoWrite + Task(子任务) + Edit~80KStep 1 输出可能被 Prune
6TodoWrite + Edit + Bash(test)~110KStep 1-2 输出被 Prune
7TodoWrite + Edit + Edit~140KStep 1-3 输出被 Prune
8TodoWrite + Bash(lint) + Edit~170K可能触发 Compaction

关键问题：旧的 TodoWrite 工具输出被 Prune 清除后，模型还能看到自己的任务列表吗？

答案：Todo 数据和上下文是分离的两套系统

机制一：Prune — 智能裁剪旧工具输出

// compaction.ts — Prune 核心算法
export const PRUNE_MINIMUM = 20_000    // 最少要裁剪 20K tokens
export const PRUNE_PROTECT = 40_000    // 保护最近 40K tokens 不被裁剪
const PRUNE_PROTECTED_TOOLS = ["skill"] // 只有 skill 工具永不裁剪

// 算法：从最新消息往回扫描
for (let msgIndex = msgs.length - 1; msgIndex >= 0; msgIndex--) {
    // 跳过最近 2 轮对话（turns < 2 的不裁剪）
    if (turns < 2) continue
    // 遇到上次 compaction 的摘要消息就停止
    if (msg.info.role === "assistant" && msg.info.summary) break

    for (let partIndex = msg.parts.length - 1; partIndex >= 0; partIndex--) {
        const part = msg.parts[partIndex]
        if (part.type === "tool" && part.state.status === "completed") {
            total += Token.estimate(part.state.output)
            if (total > PRUNE_PROTECT) {     // 超过 40K 阈值
                toPrune.push(part)           // 标记为待裁剪
            }
        }
    }
}

// 被裁剪的工具输出在 toModelMessages 中变成：
"[Old tool result content cleared]"

// 关键：TodoWrite 的输出也会被裁剪！
// 但 Todo 数据本身在 SQLite 中完好无损
Prune 调用时机：在 loop() 循环完全结束后（prompt.ts 第 716 行 SessionCompaction.prune({sessionID})），为下一次对话清理上下文。这意味着在同一个 loop 内，Prune 不会中途清除工具输出。

机制二：Compaction — 上下文溢出时的智能摘要

// 当 token 数超过模型上下文窗口时自动触发
// compaction.ts — isOverflow 检测
const count = tokens.total || tokens.input + tokens.output + cache.read + cache.write
const usable = model.limit.input - reserved  // 或 context - maxOutputTokens
return count >= usable   // true → 触发 Compaction

// Compaction 使用专门的 compaction agent 生成摘要
// 摘要模板（defaultPrompt）：
`## Goal
[What goal(s) is the user trying to accomplish?]

## Instructions
[What important instructions did the user give you?]

## Discoveries
[What notable things were learned during this conversation?]

[What work has been completed, what is in progress,
 and what work is left?]
       ↑ 这里自然包含 Todo 状态

// 摘要完成后，filterCompacted 截断历史：
// [旧消息1] [旧消息2] ... [compaction part] ← 截断点
// [摘要消息] [新消息...] ← 模型只看到这些
Compaction 的风险：摘要由 AI 生成，可能遗漏某些 Todo 项。Compaction 后模型看到的是一段文字总结，而非结构化的 Todo JSON。这就是为什么 todoread.txt 指示模型"在不确定时调用 TodoRead 刷新状态"。

机制三：TodoRead — 模型随时刷新 Todo 状态

        return { output: JSON.stringify(todos, null, 2), ... }

机制四：TodoWrite 全量替换 — 每次都是完整快照

// Todo.update() — 全量覆盖策略
export function update(input: { sessionID: string; todos: Info[] }) {
    Database.transaction((db) => {
        // ① 先删除该 session 的全部 Todo
        db.delete(TodoTable).where(eq(TodoTable.session_id, input.sessionID)).run()
        if (input.todos.length === 0) return
        // ② 再插入新列表
        db.insert(TodoTable)
            .values(input.todos.map((todo, position) => ({
                session_id: input.sessionID,
                content: todo.content,
                status: todo.status,
                priority: todo.priority,
                position,
            })))
            .run()
    })
    Bus.publish(Event.Updated, input)  // → UI 实时刷新
}

// 这意味着：
// - 模型每次调用 TodoWrite 都必须提供 完整列表（不是增量）
// - SQLite 中永远保存的是最新、最完整的任务状态
// - 即使上下文中的旧 TodoWrite 输出全部被清除
//   模型只需调用一次 TodoRead，就能恢复完整的任务地图

// ② 旧的 TodoWrite 输出（Step 1-3）被替换为 "[Old tool result content cleared]"
// ③ 但模型只需要看 最近一次 TodoWrite 的输出就知道当前状态
// ④ 如果不确定，随时调用 TodoRead 从 SQLite 刷新
// ⑤ 如果触发 Compaction，摘要会包含"已完成/进行中/剩余"信息

总结：为什么 10 步任务到第 8 步不会丢失 Todo？

  1. Prune 保留最近 40K tokens — 最近 1-2 次 TodoWrite 输出一定在保护范围内
  2. TodoWrite 全量替换 — 最近一次输出就包含完整的任务列表，不需要看旧版本
  3. SQLite 持久化 — 即使上下文全部丢失，TodoRead 一行就能恢复
  4. Compaction 摘要 — 如果触发压缩，摘要模板包含"已完成/进行中/剩余"
  5. todoread.txt 指令 — 明确要求"after every few messages"调用 TodoRead 确认状态
  6. UI 面板 — Bus 事件驱动 Todo 面板实时刷新，用户可以随时看到进度

本质上，上下文是有损的，但数据库是无损的。Todo 系统的设计哲学是：不依赖上下文来保存任务状态，而是通过 SQLite 作为"单一真相源"，上下文只是展示层。

核心规律总结

1. TodoWrite 是锚，不是锁

TodoWrite 给模型提供了一个"任务地图"，但不会强制执行。模型仍然可能跳过某些任务。真正起作用的是：Todo 列表在每轮循环中都出现在消息历史中，模型通过 pattern matching 被持续引导。

2. system-reminder 是纠偏器

当用户中途插入消息时，<system-reminder> 标签中的 "continue with your tasks" 指令确保模型不会偏离主线。这是一个"软约束"，通过 prompt engineering 实现。

3. Doom Loop 是硬刹车

Doom Loop 检测是唯一一个"硬约束"——通过代码逻辑检测重复调用，直接暂停执行。阈值设为 3 次，既不会误判正常重试，又能及时阻止死循环。

4. 子任务隔离是防火墙

子 Agent 对 TodoWrite/TodoRead 的双重 deny（permission + tools 注册）确保任务列表的"单一写者"原则——只有父 Agent 能修改自己的 Todo。

5. max-steps 是兜底

无论前面的防线是否生效，max-steps 保证循环一定会终止。到达上限后，模型被迫输出总结，让用户知道还有什么没做完。

6. Compaction 是氧气面罩

当上下文窗口溢出时，Compaction 压缩旧消息但保留关键信息（包括 Todo 状态）。这防止模型因"失忆"而重做已完成的任务或遗忘未完成的任务。

防走偏 ≠ 防犯错

以上六层机制解决的是"任务方向"问题（模型是否在做正确的事），而不是"执行质量"问题（模型是否把事做好了）。如果模型在实现 JWT 工具时写了错误的代码，TodoWrite 和 Doom Loop 都无法检测到——这需要通过测试（Bash 运行 vitest）和代码审查（委派 code-reviewer Agent）来保证。

TodoWrite 的最大价值不在于"强制模型按计划执行"，而在于让模型和用户共享同一张任务地图。当用户看到 Todo 面板上的进度时，他们可以判断模型是否在正确的方向上，并在必要时介入纠偏。
