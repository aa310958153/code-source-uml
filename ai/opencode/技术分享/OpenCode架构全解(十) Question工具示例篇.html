Question 工具存在的意义

核心问题：AI Agent 不应该"猜"用户的需求

在传统的 AI 编码助手中，当模型遇到歧义时，通常会自行做出假设并继续执行。这导致：

问题后果Question 的解决方式
需求歧义模型选错技术栈，重做结构化选项让用户选
实现分歧模型用 A 方案，用户要 B提前展示方案对比
上下文缺失模型不知道用户的业务约束主动询问关键信息
审批节点模型直接执行危险操作Plan 完成后等用户确认

Question 的设计哲学

传统 AI 助手

用户："添加认证系统"
AI：(假设用 JWT，开始写代码)
...3步后发现用户想用 Session...
→ 回退、重做、浪费 token

OpenCode Question

用户："添加认证系统"
AI：调用 Question 工具 →

  ○ JWT (无状态，适合 API)
  ○ Session (有状态，适合 Web)
→ 用户选择后，精准执行

Promise 阻塞：核心魔法

Question 如何让整个 AI 循环"暂停"？

这是 Question 工具最精妙的设计：利用 JavaScript 的 Promise 机制，让 AI 的 ReAct 循环在 await 处自然挂起，直到用户回复。

// src/question/index.ts — ask() 函数
export async function ask(input: {
    sessionID: string
    questions: Info[]
    tool?: { messageID: string; callID: string }
}): Promise<Answer[]> {
    const s = await state()
    const id = Identifier.ascending("question")

    return new Promise<Answer[]>((resolve, reject) => {
        const info: Request = {
            id, sessionID: input.sessionID,
            questions: input.questions, tool: input.tool,
        }
        // ① 将 resolve/reject 保存到 pending 字典
        s.pending[id] = { info, resolve, reject }
        // ② 通过 Bus 发布事件 → SSE → 前端
        Bus.publish(Event.Asked, info)
    })
    // ③ Promise 没有立即 resolve → await 会一直等待
    // ④ 整个 tool.execute() 卡住 → 整个 ReAct 循环卡住
}

阻塞链路图

// 调用栈（从上到下依次 await）：
prompt.ts loop()                    // ← 主循环
  └→ processor.process()            // ← 处理 LLM 流
       └→ AI SDK tool execution     // ← 执行工具
            └→ question.ts execute  // ← QuestionTool.execute()
                 └→ Question.ask()  // ← new Promise() 挂起
                      └→ ???        // 等待用户回复

// 当用户回复时：
Question.reply()                    // ← API 调用
  └→ pending[id].resolve(answers)   // ← Promise resolve
       └→ Question.ask() 返回       // ← await 恢复
            └→ execute() 返回       // ← 格式化答案
                 └→ processor 继续  // ← 工具结果传回模型
                      └→ loop 继续  // ← 下一轮 ReAct
关键理解：这不是"轮询"或"定时器"，而是原生的 Promise 异步等待。Node.js/Bun 的事件循环会在 Promise pending 时处理其他事件（如 HTTP 请求），因此服务器不会阻塞。

pending 字典 — 内存中的等待队列

// Instance.state — 进程级别的单例状态
const state = Instance.state(async () => {
    const pending: Record<string, {
        info: Request           // 问题详情
        resolve: (answers: Answer[]) => void  // 成功回调
        reject: (e: any) => void              // 失败回调
    }> = {}
    return { pending }
})

SSE 事件驱动与前端状态同步

三个 Bus 事件

事件触发时机携带数据前端行为
question.asked Question.ask() 调用时 完整 Request（id, sessionID, questions[], tool?） 添加到 store.question[sessionID] 数组
question.replied Question.reply() 调用时 { sessionID, requestID, answers[] } 从 store 中移除该 request
question.rejected Question.reject() 调用时 { sessionID, requestID } 从 store 中移除该 request

前端 Event Reducer — 状态同步

// packages/app/src/context/global-sync/event-reducer.ts
case "question.asked": {
    const question = event.properties as QuestionRequest
    const questions = input.store.question[question.sessionID]
    if (!questions) {
        // 首次提问：创建数组
        input.setStore("question", question.sessionID, [question])
        break
    }
    // 已有提问：二分查找插入位置，保持有序
    const result = Binary.search(questions, question.id, (q) => q.id)
    if (result.found) {
        input.setStore("question", question.sessionID, result.index, reconcile(question))
    } else {
        input.setStore("question", question.sessionID,
            produce((draft) => { draft.splice(result.index, 0, question) }))
    }
    break
}

case "question.replied":
case "question.rejected": {
    // 回复或拒绝 → 从 store 中移除
    const props = event.properties as { sessionID: string; requestID: string }
    const questions = input.store.question[props.sessionID]
    if (!questions) break
    const result = Binary.search(questions, props.requestID, (q) => q.id)
    if (!result.found) break
    input.setStore("question", props.sessionID,
        produce((draft) => { draft.splice(result.index, 1) }))
    break
}

// 模块级缓存 — 组件卸载时保存，重新挂载时恢复
const cache = new Map<string, {
    tab: number; answers: QuestionAnswer[];
    custom: string[]; customOn: boolean[];
}>()

// onCleanup 时保存到 cache
onCleanup(() => {
    if (replied) return
    cache.set(props.request.id, {
        tab: store.tab,
        answers: store.answers.map((a) => (a ? [...a] : [])),
        custom: store.custom.map((s) => s ?? ""),
        customOn: store.customOn.map((b) => b ?? false),
    })

// 初始化时从 cache 恢复

    tab: cached?.tab ?? 0,
    answers: cached?.answers ?? [],

RejectedError 与循环中断

用户点 Dismiss 时会发生什么？

// src/question/index.ts — reject 流程
export async function reject(requestID: string): Promise<void> {
    const s = await state()
    const existing = s.pending[requestID]
    if (!existing) return

    delete s.pending[requestID]
    Bus.publish(Event.Rejected, { sessionID, requestID })

    existing.reject(new RejectedError())
    //              ↑ Promise reject → 抛出异常
}

export class RejectedError extends Error {
    constructor() {
        super("The user dismissed this question")
    }
}

Processor 如何捕获 RejectedError？

// src/session/processor.ts — tool-error 分支
case "tool-error": {
    const match = toolcalls[value.toolCallId]
    if (match && match.state.status === "running") {
        await Session.updatePart({
            ...match,
            state: {
                status: "error",
                error: (value.error as any).toString(),
                // ...
            },
        })

        // 关键：区分用户拒绝 vs 真正的错误
        if (
            value.error instanceof PermissionNext.RejectedError ||
            value.error instanceof Question.RejectedError
        ) {
            blocked = shouldBreak
            // shouldBreak = config.experimental?.continue_loop_on_deny !== true
            // 默认 true → 循环中断，AI 停止
        }
        delete toolcalls[value.toolCallId]
    }
    break
}
默认行为：用户拒绝提问后，blocked = true → processor 的 while(true) 循环 break → AI 停止输出。这符合直觉：用户拒绝了，AI 应该停下来等用户进一步指示。

Headless 模式下的 deny 策略

CI/CD 中 Question = 死锁

在没有交互式 UI 的环境中（GitHub Actions、opencode run），Question.ask() 的 Promise 永远不会 resolve，导致 AI 循环永久挂起。解决方案：在权限层直接 deny。

// src/cli/cmd/run.ts — opencode run 命令
const rules: PermissionNext.Ruleset = [
    {
        permission: "question",
        action: "deny",        // ← 直接拒绝
        pattern: "*",
    },
    {
        permission: "plan_enter",
        action: "deny",
        pattern: "*",
    },
    {
        permission: "plan_exit",
        action: "deny",
        pattern: "*",
    },
]

// 创建 session 时传入 deny 规则
const result = await sdk.session.create({
    title: name,
    permission: rules,  // ← session 级别的权限覆盖
})

GitHub Actions — 同样 deny

// src/cli/cmd/github.ts — GitHub Actions 处理
session = await Session.create({
    permission: [
        {
            permission: "question",
            action: "deny",
            pattern: "*",
        },
    ],
})

// 效果：模型调用 Question 工具时
// → PermissionNext.ask() 检测到 deny 规则
// → 抛出 RejectedError
// → 模型收到 "The user dismissed this question"
// → 模型自行做出合理假设继续执行
优雅降级：deny 后模型不会崩溃，而是收到 "dismissed" 错误，自行做出合理假设继续执行。这是 Question 工具的 fallback 设计 — 在无法提问的环境中，AI 退化为"自主决策"模式。

    permission: [
        { permission: "todowrite", pattern: "*", action: "deny" },
        { permission: "todoread",  pattern: "*", action: "deny" },
        // 注意：question 没有在 deny 列表中！
        // → 子 Agent 可以使用 Question 工具向用户提问
    ],
})

// 但工具注册层：
tools: {
    todowrite: false,
    todoread: false,
    // question 没有显式禁用 → 子 Agent 可以调用
}
设计意图：子 Agent 可能需要向用户确认技术细节（例如 explore agent 发现多种可行方案时）。因此 Question 权限对子 Agent 开放，但 TodoWrite 权限关闭（防止子 Agent 篡改父 Agent 的任务列表）。

完整生命周期与核心规律总结

Question 工具的完整生命周期

1. Promise 是核心原语

不用轮询、不用 WebSocket、不用定时器。一个 new Promise 就能让整个 AI 循环优雅地暂停。resolve/reject 存储在内存字典中，通过 API 路由触发。

2. 事件驱动的双向通信

Bus 事件 → SSE → 前端状态同步。前端不直接调用后端轮询，而是订阅 SSE 流。question.asked 到达时渲染 UI，replied/rejected 到达时移除 UI。

3. 客户端感知注册

不是所有客户端都能用 Question。registry.ts 根据 OPENCODE_CLIENT 类型条件注册。没有交互 UI 的环境（CI/CD、API 直调）会被 deny，AI 退化为自主决策。

4. 结构化 > 自由文本

Question 输出是 label + description 的选项列表，而非让用户自由输入。这减少了歧义、提高了回答质量。custom: true 保留灵活性但作为"兜底"而非主要输入方式。

5. 复用基础设施

PlanExitTool 复用 Question.ask() 实现审批流程。Question 不仅是一个"提问工具"，更是一个通用的"AI 暂停并等待用户输入"的基础设施。

6. 优雅降级

当 Question 不可用（deny）或被用户拒绝（reject）时，AI 不会崩溃。RejectedError 被 processor 捕获，AI 收到 "dismissed" 错误后自行决定下一步。
