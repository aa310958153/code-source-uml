一、为什么需要多 Agent？

在单体 Agent 架构中，一个 AI 需要同时承担「分析代码」「执行修改」「搜索文件」「规划方案」等所有职责。这带来三个核心问题：

问题一：权限过宽

分析型任务（搜索代码、理解架构）本不需要写权限，但单体 Agent 同时拥有读写权限，增加了误操作风险。

问题二：上下文污染

当 AI 同时进行规划和实现时，规划阶段的讨论会占用实现阶段的上下文窗口，导致长任务中上下文溢出。

问题三：缺乏专业化

不同任务需要不同的 system prompt 和工具集。代码搜索 Agent 需要 grep/glob，而构建 Agent 需要 edit/bash。

OpenCode 的解决方案是 多 Agent 系统：将职责拆分为独立 Agent，每个 Agent 拥有独立的权限集、工具集和 system prompt，通过 TaskTool 进行编排。

二、Agent 系统全景架构

三、Agent.Info — Zod Schema 数据模型

每个 Agent 的配置由 Agent.Info Zod Schema 定义，这是整个 Agent 系统的数据基础：

// packages/opencode/src/agent/agent.ts
export namespace Agent {
    export const Info = z.object({
        name: z.string(),                          // Agent 唯一标识
        description: z.string().optional(),         // 描述（模型看到的）
        mode: z.enum(["subagent", "primary", "all"]),  // 角色分类
        native: z.boolean().optional(),             // 是否内置
        hidden: z.boolean().optional(),             // 是否对用户隐藏
        topP: z.number().optional(),                // 模型采样参数
        temperature: z.number().optional(),
        color: z.string().optional(),               // UI 颜色标识
        permission: PermissionNext.Ruleset,         // 权限规则集（核心）
        model: z.object({                           // 指定模型（可选）
            modelID: z.string(),
            providerID: z.string(),
        }).optional(),
        variant: z.string().optional(),             // Prompt 变体
        prompt: z.string().optional(),              // 自定义 system prompt
        options: z.record(z.string(), z.any()),     // 模型特定选项
        steps: z.number().int().positive().optional(),  // 最大步数限制
    })
}
关键字段：permission 是每个 Agent 的核心差异 — build 拥有全权限，plan 禁止所有编辑操作，explore 只保留搜索类工具。权限决定了 Agent 的"能力边界"。

四、七种内置 Agent 详解

planprimary否禁止所有 edit，仅允许写 .opencode/plans/*.md只读分析和规划
generalsubagent否全权限但 deny todoread/todowrite通用研究和多步任务

            grep: "allow",           // 只开搜索

            list: "allow",
            bash: "allow",

explore Agent 的设计体现了一个重要原则：最小权限。它只能搜索和阅读，不能编辑任何文件。这使得它可以安全地被频繁调用，不会造成代码破坏。

五、权限隔离三层架构

每个 Agent 的最终权限由三层合并决定：

defaults (基础层)
→
agentRules (Agent 层)
→
user (用户层)
// 第一层：defaults — 所有 Agent 共享的基础权限
const defaults = PermissionNext.fromConfig({
    "*": "allow",                      // 默认允许所有工具
    doom_loop: "ask",                  // 死循环检测需确认
    external_directory: {"*": "ask"},  // 外部目录需确认
    question: "deny",                  // 默认禁止提问
    plan_enter: "deny",                // 默认禁止进入 plan
    plan_exit: "deny",                 // 默认禁止退出 plan
    read: {
        "*": "allow",
        "*.env": "ask",               // .env 文件需确认
        "*.env.*": "ask",
        "*.env.example": "allow",
    },
})

// 第二层：agentRules — 每个 Agent 的特殊规则
// 例如 build Agent 开放 question 和 plan_enter
PermissionNext.fromConfig({ question: "allow", plan_enter: "allow" })

// 第三层：user — 用户在 opencode.json 中的配置
const user = PermissionNext.fromConfig(cfg.permission ?? {})

// 最终合并（后者覆盖前者）
PermissionNext.merge(defaults, agentRules, user)
合并优先级：user > agentRules > defaults。这意味着用户可以通过 opencode.json 覆盖任何 Agent 的权限设置。但要注意，PermissionNext.merge 的行为是"后面的规则覆盖前面的同权限同模式规则"。

六、Agent Mode — 角色分类机制

Agent 的 mode 字段决定了它在系统中的角色定位：

Mode含义对用户可见可被 TaskTool 调用可设为主 Agent
primary主 Agent，直接响应用户是否是
subagent子 Agent，由 TaskTool 编排是是否
all两者皆可是是是
// TaskTool 中过滤可用 Agent 的逻辑
const agents = await Agent.list()
    .then(x => x.filter(a => a.mode !== "primary"))
// 只有 mode !== "primary" 的 Agent 才能作为子 Agent 被调用

// defaultAgent 中的过滤
const primaryVisible = Object.values(agents)
    .find(a => a.mode !== "subagent" && a.hidden !== true)
// 只有 mode !== "subagent" 且非隐藏的 Agent 才能作为默认 Agent
hidden 字段的作用：compaction、title、summary 三个 Agent 虽然 mode 是 "primary"，但 hidden=true 使它们对用户不可见。它们是系统内部的"幕后工作者"，在需要时被自动调用。

七、TaskTool — 子 Agent 编排引擎

TaskTool 是多 Agent 系统的编排核心，它负责创建子 Session、启动子 Agent 的 ReAct 循环，并将结果返回给主 Agent。

7.1 工具定义

// packages/opencode/src/tool/task.ts
export const TaskTool = Tool.define("task", async (ctx) => {
    // 动态生成可用 Agent 列表
    const agents = await Agent.list()
        .then(x => x.filter(a => a.mode !== "primary"))

    // 按权限过滤（调用者可能无权访问某些 Agent）
    const caller = ctx?.agent
    const accessible = caller
        ? agents.filter(a => PermissionNext.evaluate("task", a.name, caller.permission).action !== "deny")
        : agents

    // 将 Agent 列表注入工具描述
    const description = DESCRIPTION.replace(
        "{agents}",
        accessible.map(a => `- ${a.name}: ${a.description}`).join("\n"),
    )
    return { description, parameters, async execute(params, ctx) { ... } }
})

7.2 执行流程

1
权限检查 — 通过 ctx.ask() 请求 task 权限
2
创建子 Session — Session.create({ parentID: ctx.sessionID })
3
设置权限 — 子 Session deny todoread/todowrite，防止子 Agent 修改主任务列表
4
启动 ReAct 循环 — SessionPrompt.prompt({ sessionID: session.id })
5
返回结果 — 将子 Session 最后一个 text part 包装为 <task_result> 返回

7.3 关键实现细节

// 子 Session 创建时的权限注入
const session = await Session.create({
    parentID: ctx.sessionID,
    title: params.description + ` (@${agent.name} subagent)`,
    permission: [
        { permission: "todowrite", pattern: "*", action: "deny" },  // 禁止写 Todo
        { permission: "todoread", pattern: "*", action: "deny" },   // 禁止读 Todo
        ...(hasTaskPermission ? [] : [
            { permission: "task", pattern: "*", action: "deny" },   // 防止嵌套子 Agent
        ]),
    ],
})

// 结果返回格式
const output = [
    `task_id: ${session.id} (for resuming to continue this task if needed)`,
    "",
    "<task_result>",
    text,
    "</task_result>",
].join("\n")

task_id — 会话恢复机制

TaskTool 支持 task_id 参数，允许主 Agent 恢复之前的子 Session 继续对话，而非每次创建新 Session。这对于需要多轮交互的研究任务非常重要。

// 如果有 task_id 且 session 存在，则复用
if (params.task_id) {
    const found = await Session.get(params.task_id).catch(() => {})
    if (found) return found   // 复用已有 session
}
// 否则创建新 session
return await Session.create({ parentID: ctx.sessionID, ... })

八、子 Session 隔离与权限继承

子 Agent 运行在独立的子 Session 中，通过 parentID 与父 Session 关联：

隔离机制

维度父 Session子 Session
消息历史完整的对话历史只有新的 prompt
Todo 工具可用deny（防止干扰主任务列表）
Agent 权限build 全权限Agent 定义中的权限 + session.permission
模型用户选择的模型Agent 配置的模型 或 继承父模型
中止控制父 AbortController监听父 abort 事件，联动取消
// 中止联动 — 父 Agent 取消时自动取消子 Agent
function cancel() {
    SessionPrompt.cancel(session.id)
}
ctx.abort.addEventListener("abort", cancel)
using _ = defer(() => ctx.abort.removeEventListener("abort", cancel))
AbortController 联动：子 Agent 的执行通过事件监听与父 Agent 绑定。当用户在 UI 中点击"Stop"时，父 AbortController.abort() 会级联取消所有正在运行的子 Agent，防止资源泄漏。

九、Plan 模式 — 只读分析 Agent

Plan Agent 是 OpenCode 双模式（Plan/Build）设计的核心。它拥有完整的阅读和分析能力，但禁止所有代码修改。

plan: {
    name: "plan",
    description: "Plan mode. Disallows all edit tools.",
    permission: PermissionNext.merge(
        defaults,
        PermissionNext.fromConfig({
            question: "allow",       // 可以向用户提问
            plan_exit: "allow",      // 可以退出 plan 模式
            external_directory: {
                [path.join(Global.Path.data, "plans", "*")]: "allow",
            },
            edit: {
                "*": "deny",                                        // 禁止所有编辑
                [path.join(".opencode", "plans", "*.md")]: "allow", // 只允许写计划文件
            },
        }),
        user,
    ),
    mode: "primary",
    native: true,
}

Plan 文件路径约定

// 有 VCS（Git）的项目：计划文件存放在项目内
path.join(Instance.worktree, ".opencode", "plans", ...)

// 无 VCS 的项目：计划文件存放在全局数据目录
path.join(Global.Path.data, "plans", ...)

Plan 模式允许编辑的唯一文件是 .opencode/plans/*.md — 这使得 AI 可以在 plan 阶段将分析结论写入 Markdown 文件，然后由 Build Agent 参考实施。

十、Plan→Build 切换机制

Plan 到 Build 的切换通过 PlanExitTool 实现，它巧妙地复用了 Question 工具的 Promise 阻塞机制：

// packages/opencode/src/tool/plan.ts
export const PlanExitTool = Tool.define("plan_exit", {
    async execute(_params, ctx) {
        // 复用 Question.ask() 向用户确认
        const answers = await Question.ask({
            sessionID: ctx.sessionID,
            questions: [{
                question: `Plan at ${plan} is complete. Would you like to switch to the build agent?`,
                header: "Build Agent",
                custom: false,
                options: [
                    { label: "Yes", description: "Switch to build agent and start implementing" },
                    { label: "No", description: "Stay with plan agent to continue refining" },
                ],
            }],
        })
        const answer = answers[0]?.[0]
        if (answer === "No")
            throw new Question.RejectedError()  // 用户选择"否"则中断

        // 创建 build Agent 的用户消息，触发模式切换
        const id = Identifier.ascending("message")
        await Session.updateMessage({
            id,
            sessionID: ctx.sessionID,
            role: "user",
            agent: "build",         // 关键：切换到 build Agent
            model: lastUser.model,
        })
        // 注入 plan 文件内容作为上下文
        await Session.updatePart({
            id: Identifier.ascending("part"),
            messageID: id,
            sessionID: ctx.sessionID,
            type: "text",
            text: BUILD_SWITCH.replace("{plan}", planContent),
        })
    },
})

十一、自定义 Agent — 配置驱动的扩展

用户可以在 opencode.json 中定义自定义 Agent，系统会在初始化时将其合并到 Agent Registry：

// opencode.json
{
  "agent": {
    "reviewer": {
      "description": "Code review specialist",
      "model": "anthropic/claude-sonnet-4-20250514",
      "prompt": "You are a code review expert...",
      "mode": "subagent",
      "permission": {
        "edit": { "*": "deny" },
        "bash": { "*": "deny" }
      }
    },
    "build": {
      // 修改内置 build Agent 的配置
      "steps": 50
    },
    "explore": {
      "disable": true   // 禁用内置 explore Agent
    }
  }
}

配置加载逻辑

// agent.ts state() 中的配置合并
for (const [key, value] of Object.entries(cfg.agent ?? {})) {
    if (value.disable) {
        delete result[key]          // 禁用已有 Agent
        continue
    }
    let item = result[key]
    if (!item)
        item = result[key] = {      // 创建全新 Agent
            name: key,
            mode: "all",
            permission: PermissionNext.merge(defaults, user),
            options: {},
            native: false,
        }
    // 合并各项配置
    item.model = value.model ?? item.model
    item.prompt = value.prompt ?? item.prompt
    item.steps = value.steps ?? item.steps
    item.permission = PermissionNext.merge(
        item.permission,
        PermissionNext.fromConfig(value.permission ?? {})
    )
}
注意：自定义 Agent 的权限是在 defaults + user 基础上追加，而不是替换。如果你希望一个自定义 Agent 只有读权限，需要显式设置 "edit": {"*": "deny"}。

十二、Agent.generate — AI 辅助创建 Agent

OpenCode 还提供了一个用 AI 生成 Agent 配置的功能：

// agent.ts
export async function generate(input: {
    description: string
    model?: { providerID: string; modelID: string }
}) {
    const model = await Provider.getModel(...)
    const language = await Provider.getLanguage(model)

    const result = await generateObject({
        model: language,
        temperature: 0.3,
        messages: [
            { role: "system", content: PROMPT_GENERATE },
            { role: "user", content: `Create an agent configuration based on: "${input.description}"
              IMPORTANT: These already exist: ${existing.map(i => i.name).join(", ")}` },
        ],
        schema: z.object({
            identifier: z.string(),
            whenToUse: z.string(),
            systemPrompt: z.string(),
        }),
    })
    return result.object
}

这个功能使用 generateObject（AI SDK 的结构化输出）让 AI 根据描述生成 Agent 的 identifier、使用时机和 system prompt，然后用户可以将其写入 opencode.json。

十三、Doom Loop 检测与防护

在 processor.ts 的 ReAct 循环中，OpenCode 实现了一个精巧的"死循环检测"机制：

// session/processor.ts
const DOOM_LOOP_THRESHOLD = 3

// 在 tool-call 事件中检测
case "tool-call": {
    const parts = await MessageV2.parts(input.assistantMessage.id)
    const lastThree = parts.slice(-DOOM_LOOP_THRESHOLD)

    if (
        lastThree.length === DOOM_LOOP_THRESHOLD &&
        lastThree.every(p =>
            p.type === "tool" &&
            p.tool === value.toolName &&                    // 同一个工具
            p.state.status !== "pending" &&
            JSON.stringify(p.state.input) === JSON.stringify(value.input)  // 相同参数
        )
    ) {
        // 触发 doom_loop 权限请求
        await PermissionNext.ask({
            permission: "doom_loop",
            patterns: [value.toolName],
            sessionID: input.assistantMessage.sessionID,
            metadata: { tool: value.toolName, input: value.input },
            always: [value.toolName],
            ruleset: agent.permission,
        })
    }
}
触发条件：当模型连续 3 次调用同一个工具且参数完全相同时，系统认为进入了死循环。此时会触发 doom_loop 权限请求，让用户决定是否继续。默认配置中 doom_loop 是 "ask"，需要用户确认。

十四、完整生命周期总结

Agent 系统核心规律

规律体现
权限即能力边界每个 Agent 的 permission 精确控制它能使用的工具，plan 不能 edit，explore 只能搜索
Session 隔离子 Agent 运行在独立的子 Session 中，消息历史、Todo 状态完全隔离
AbortController 联动父子 Agent 通过 abort 事件绑定，一键停止全部取消
动态工具描述TaskTool 的描述中动态注入可用 Agent 列表，模型据此选择调用哪个子 Agent
三层权限合并defaults（基础）→ agent（Agent 特有）→ user（用户覆盖），后者覆盖前者
Question 复用PlanExitTool 复用 Question.ask() 实现 Plan→Build 审批，统一的人机交互原语
配置驱动opencode.json 可以创建新 Agent、修改内置 Agent、甚至禁用内置 Agent
task_id 会话恢复子 Agent Session 可以被恢复继续，实现多轮交互而非一次性任务
