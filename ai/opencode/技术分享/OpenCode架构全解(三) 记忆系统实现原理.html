一、核心洞察：LLM 记忆的本质

为什么 OpenCode 不用向量数据库？

向量数据库做记忆检索存在根本问题：LLM 需要的是结构化、时序化的上下文，而不是语义相似性检索。
"我们做了什么、改了哪些文件、下一步干什么"——这些信息用对话历史自然表达，向量检索反而丢失了时序和因果关系。

OpenCode 的选择：让对话历史本身就是记忆。当它太长时，用 LLM 自己压缩成摘要（摘要也是对话的一部分）。模型天然能理解"之前的摘要 + 最近的对话"。

二、记忆三层架构

三层架构的精妙之处：每一层用不同的技术，解决不同的问题——
• 长期记忆用文件解决"跨会话知识"（代码规范、项目约定）
• 中期记忆用LLM 摘要解决"对话太长"（之前的工作总结）
• 短期记忆用原始消息解决"当前工作细节"（最近读的文件、写的代码）

三、五大机制与介入时序

OpenCode 的 Session Loop 是一个 while(true) 循环，每轮循环代表一次 LLM 交互。5 个记忆机制在循环的不同时间点介入：

机制核心文件触发时机是否调 LLM目的
filterCompacted message-v2.ts:814 每轮 loop 开始 否 跳过摘要前的旧消息
Instruction instruction.ts 构建 system prompt 否 注入跨会话持久知识
Summary summary.ts step === 1 时异步 否 计算代码变更 diff
Compaction compaction.ts LLM 返回后检查 是 用 LLM 将旧对话压缩为摘要
Prune compaction.ts:58 整个 loop 结束 否 标记远古工具输出为已清理

四、机制一：Instruction — 长期记忆注入

4.1 工作原理

Instruction 是 OpenCode 的"长期记忆"——它从文件系统中读取持久化的指令文件，注入到每轮 LLM 调用的 System Prompt 中。这样即使跨会话，Agent 也能"记住"项目的编码规范、架构约定等。

1
扫描指令文件路径
instruction.ts → systemPaths()
按优先级查找：项目级 AGENTS.md → CLAUDE.md → CONTEXT.md（deprecated）
然后查找全局级：~/.config/opencode/AGENTS.md → ~/.claude/CLAUDE.md
最后查找 config.instructions 中的自定义路径和 URL。
2
读取文件内容
instruction.ts → system()
读取所有找到的文件内容，HTTP URL 也会 fetch（5秒超时）。每段内容前缀 "Instructions from: {path}" 让 LLM 知道来源。
3
注入 System Prompt
prompt.ts:652
与 SystemPrompt.environment() 合并，形成完整的 system prompt：
const system = [...SystemPrompt.environment(model), ...InstructionPrompt.system()]

4.2 智能子目录指令发现

Instruction 还有一个精巧的子目录发现机制：当 LLM 读取某个文件时，resolve() 会自动向上扫描目录树，找到该文件所在子目录的 AGENTS.md：

// instruction.ts:168-191
// 当 Agent 读取 src/db/users.ts 时，自动加载：
//   src/db/AGENTS.md     (如果存在)
//   src/AGENTS.md        (如果存在)
// 但跳过 systemPaths() 已加载的全局指令

export async function resolve(messages, filepath, messageID) {
    const system = await systemPaths()       // 已加载的全局指令
    const already = loaded(messages)          // 历史中已读取的文件
    const results = []

    let current = path.dirname(filepath)
    const root = path.resolve(Instance.directory)

    while (current.startsWith(root) && current !== root) {
        const found = await find(current)     // 查找 AGENTS.md / CLAUDE.md
        if (found && !system.has(found) && !already.has(found)
            && !isClaimed(messageID, found)) {
            claim(messageID, found)            // 防重复注入
            results.push({ filepath: found, content: ... })
        }
        current = path.dirname(current)       // 向上遍历
    }
    return results
}
claim 机制的精妙之处：
• isClaimed(messageID, filepath) 防止同一条指令在同一轮对话中被重复注入
• loaded(messages) 检查历史中已被 Read 工具读取过的指令文件，避免冗余
• clear(messageID) 在每轮 LLM 调用结束后清理 claim 记录
→ 按需加载 + 去重保护，而不是暴力全量注入

五、机制二：filterCompacted — 记忆回放

5.1 核心问题

当对话很长并经过多次 Compaction 后，DB 中存储了数百条消息。如果全部加载发送给 LLM，既浪费 token 又包含过时信息。需要一种方式："从最近的摘要开始，只加载摘要之后的消息"。

5.2 算法实现

// message-v2.ts:814-830

export async function filterCompacted(stream) {
    const result = []
    const completed = new Set<string>()

    for await (const msg of stream) {           // stream 是倒序的（最新→最旧）
        result.push(msg)

        // 遇到已完成的 Compaction 请求 → 停止！
        if (msg.info.role === "user"
            && completed.has(msg.info.id)
            && msg.parts.some(part => part.type === "compaction"))
            break                               // ← 这就是"记忆边界"

        // 记录成功完成的 Compaction
        if (msg.info.role === "assistant"
            && msg.info.summary              // 这是 Compaction 生成的摘要
            && msg.info.finish               // 生成完成了
            && !msg.info.error)              // 没有出错
            completed.add(msg.info.parentID) // 记录对应的请求 ID
    }

    result.reverse()                          // 恢复为正序（旧→新）
    return result
}
设计精妙点：
• 利用 parentID 关联 User 消息和 Compaction 摘要，精准定位"记忆边界"
• summary 标志位 + finish + !error 三重条件确保只信任成功完成的压缩
• 倒序遍历 → 遇到边界 break → reverse() 恢复正序，简洁高效
• 效果：无论历史多长，只加载"最近的摘要 + 之后的原始消息"

六、机制三：Compaction — 智能压缩

Compaction 是记忆系统中最"重"的机制——它是唯一需要调用 LLM 的。当上下文窗口即将溢出时，让 LLM 自己总结之前的对话，生成结构化摘要。

6.1 触发条件

// compaction.ts:30-48

const COMPACTION_BUFFER = 20_000  // 预留 20K token 缓冲

export async function isOverflow(input) {
    const config = await Config.get()
    if (config.compaction?.auto === false) return false  // 可关闭

    const count = input.tokens.total
        || input.tokens.input + input.tokens.output
           + input.tokens.cache.read + input.tokens.cache.write

    const reserved = config.compaction?.reserved
        ?? Math.min(COMPACTION_BUFFER, maxOutputTokens(model))

    const usable = input.model.limit.input
        ? input.model.limit.input - reserved
        : context - maxOutputTokens(model)

    return count >= usable  // token 用量 >= 可用上限？
}

6.2 Compaction 摘要模板

当触发 Compaction 时，系统向 LLM 发送一段精心设计的提示词，指导它生成结构化摘要：

// compaction.ts:173-199 — 默认 Compaction Prompt

"Provide a detailed prompt for continuing our conversation above.
Focus on information that would be helpful for continuing the conversation,
including what we did, what we're doing, which files we're working on,
and what we're going to do next.

The summary that you construct will be used so that another agent can read
it and continue the work.

When constructing the summary, try to stick to this template:
---
## Goal
[What goal(s) is the user trying to accomplish?]

## Instructions
- [What important instructions did the user give you]
- [If there is a plan or spec, include information about it]

## Discoveries
[What notable things were learned during this conversation]

## Accomplished
[What work has been completed, in progress, and left?]

## Relevant files / directories
[Structured list of relevant files]
---"

6.3 完整的 Compaction 流程

1
检测溢出
isOverflow() 检查最后一轮 LLM 返回的 token 用量。如果接近上限 → 创建 Compaction 请求。
2
创建 Compaction 消息
compaction.ts:296-327 → create()
写入一条 User 消息，包含 CompactionPart（类型标记，不是实际文本）。这条消息在 toModelMessages 中会被转换为 "What did we do so far?"。
3
使用 compaction Agent 生成摘要
compaction.ts:131-166
获取 compaction Agent（专门的角色配置），创建摘要消息。
发送全部历史消息 + Compaction Prompt → LLM 生成结构化摘要。
4
自动续传（Auto-Continue）
compaction.ts:235-289
压缩完成后，如果是自动触发的（auto: true），系统自动追加一条 User 消息：
"Continue if you have next steps, or stop and ask for clarification if you are unsure how to proceed."
→ LLM 读到摘要 + 这条指令后，会自然地继续未完成的工作。
5
溢出恢复（Overflow Recovery）
compaction.ts:112-128
如果 Compaction 本身也溢出（对话太长了！），系统会尝试去掉最后一条用户消息，
从更早的位置开始压缩。还会移除媒体附件（图片/PDF）减少 token 用量。
Compaction 的优雅之处：
• 用 LLM 总结 LLM：不是简单的截断或丢弃，而是让模型自己提炼结构化信息
• 摘要也是对话：压缩结果作为一条 assistant 消息存入 DB，filterCompacted 自然能找到它
• 模板引导：Goal → Instructions → Discoveries → Accomplished → Files，确保不丢关键信息
• 自动续传：压缩后 LLM 不会"发呆"，会自动继续未完成的任务
• 插件扩展：Plugin.trigger("experimental.session.compacting") 允许插件自定义压缩逻辑

七、机制四：Prune — 工具输出裁剪

Prune 是一种廉价的、不调用 LLM 的记忆管理策略。它不做复杂的总结，只是简单地把远古的工具输出标记为"已清理"。

7.1 算法

// compaction.ts:50-98

const PRUNE_MINIMUM  = 20_000   // 最少要裁剪 20K token
const PRUNE_PROTECT  = 40_000   // 保护最近的 40K token 不裁剪
const PRUNE_PROTECTED_TOOLS = ["skill"]  // skill 工具永不裁剪

export async function prune(input) {
    const msgs = await Session.messages({ sessionID })
    let total = 0
    const toPrune = []
    let turns = 0

    // 从最新消息向旧消息遍历
    for (let i = msgs.length - 1; i >= 0; i--) {
        const msg = msgs[i]
        if (msg.info.role === "user") turns++
        if (turns < 2) continue          // 跳过最近的 2 轮对话
        if (msg.info.role === "assistant" && msg.info.summary) break  // 遇到摘要停止

        for (let j = msg.parts.length - 1; j >= 0; j--) {
            const part = msg.parts[j]
            if (part.type === "tool" && part.state.status === "completed") {
                if (PRUNE_PROTECTED_TOOLS.includes(part.tool)) continue  // 受保护工具跳过
                if (part.state.time.compacted) break  // 已裁剪的停止

                const estimate = Token.estimate(part.state.output)
                total += estimate
                if (total > PRUNE_PROTECT)     // 超过保护阈值
                    toPrune.push(part)          // → 标记裁剪
            }
        }
    }

    // 只有裁剪量 > 20K 才执行（避免无意义的微小裁剪）
    if (pruned > PRUNE_MINIMUM) {
        for (const part of toPrune) {
            part.state.time.compacted = Date.now()  // ← 打上时间戳
            await Session.updatePart(part)
        }
    }
}

7.2 裁剪效果

被裁剪的工具输出不会从 DB 中删除，但在发送给 LLM 时被替换为占位文本：

// message-v2.ts:637-638 — toModelMessages 中的处理

const outputText = part.state.time.compacted
    ? "[Old tool result content cleared]"   // ← 裁剪后
    : part.state.output                        // ← 未裁剪

const attachments = part.state.time.compacted
    ? []                                       // 裁剪后附件也清除
    : (part.state.attachments ?? [])
Prune 的设计哲学：空间换时间
• 不调用 LLM：零成本，纯数据库操作
• 保护最近 40K token：刚用的工具输出可能还有参考价值
• 保护 skill 工具：技能加载信息不应丢失
• 20K 最低阈值：太小的裁剪不值得执行
• 标记而非删除：time.compacted 时间戳可以追溯

八、机制五：Summary — 代码变更追踪

Summary 不是"记忆"本身，但它是记忆系统的辅助层——它追踪每个 Session 中实际发生了哪些代码变更，让用户和 Agent 都能快速回顾"我们改了什么"。

8.1 工作原理

// summary.ts:69-98

export const summarize = fn(..., async (input) => {
    const all = await Session.messages({ sessionID })

    await Promise.all([
        summarizeSession({ sessionID, messages: all }),    // 整个 session 的 diff
        summarizeMessage({ messageID, messages: all }),    // 单条消息的 diff
    ])
})

async function summarizeSession(input) {
    const diffs = await computeDiff({ messages })   // 从 snapshot 计算 git diff
    await Session.setSummary({
        sessionID,
        summary: {
            additions: diffs.reduce((sum, x) => sum + x.additions, 0),
            deletions: diffs.reduce((sum, x) => sum + x.deletions, 0),
            files: diffs.length,
        },
    })
}

// computeDiff: 找到最早的 step-start snapshot 和最新的 step-finish snapshot
// 调用 Snapshot.diffFull(from, to) 计算完整 diff
Summary 的价值：
• 用户可以在 UI 上看到"本次会话改了哪些文件、加了多少行、删了多少行"
• 每条用户消息也关联了 diff，可以看到"这条指令引发了哪些变更"
• 利用 Git Snapshot 机制，不依赖 LLM，零成本精确计算

九、五大机制的协作流程

以下是一个完整 Session 生命周期中，5 个机制如何协作的时序图：

协作的精妙：每个机制负责一个时间尺度
• 短期（当前轮）：原始消息 + 工具输出 → 全精度保留
• 中短期（Session 内历史轮）：Prune 裁剪远古工具输出 → 保留摘要
• 中期（Session 内跨轮）：Compaction 压缩旧对话 → filterCompacted 加载摘要
• 长期（跨 Session）：AGENTS.md 持久注入 → 项目规范永不丢失

十、数据模型：Message 与 Part

理解记忆系统的关键是理解 OpenCode 的消息数据模型。每条消息由 Info（元数据）和 Parts（内容片段）组成：

Part 类型角色记忆系统中的作用

十一、设计优势深度分析

🎯

零额外基础设施

不需要向量数据库、Redis、独立的记忆服务。所有记忆都存在 SQLite + 文件系统中，部署简单、运维成本为零。

🔄

天然兼容 LLM 理解方式

摘要是自然语言，模型天然能理解"之前的总结 + 最近的消息"。不需要 embedding 检索、相似度计算等额外步骤。

⚡

4/5 机制零 LLM 成本

filterCompacted、Instruction、Summary、Prune 都不调用 LLM。只有 Compaction 需要额外 LLM 调用，且仅在上下文快溢出时触发。

🔗

因果时序完整保留

对话历史天然保留因果关系："因为用户说了 A，所以 Agent 做了 B"。向量检索可能丢失这种时序和因果。

📊

分层降级策略

Prune（廉价）→ Compaction（中等）→ Overflow Recovery（昂贵）。先用便宜方案尝试，不行再用贵的。成本可控。

🔌

插件可扩展

Compaction Prompt 可被插件覆盖（Plugin.trigger），Instruction 支持自定义 URL/文件，灵活适配不同场景。

🛡️

安全边界清晰

Compaction 有 20K buffer 预留防溢出；Prune 保护最近 40K + skill 工具；filterCompacted 三重条件验证。层层设防。

🎭

对用户完全透明

用户感知不到 Compaction/Prune 的存在。对话看起来是连续的，Agent 似乎一直"记住"之前的事。无感的记忆管理。

分层降级的成本对比

机制 LLM 调用 触发频率 释放空间 信息损失 成本
Prune 无 每次 loop 结束 远古工具输出 完全丢失 $0
Compaction 1 次 上下文接近上限时 旧对话 → 摘要 压缩为结构化总结 ~$0.01-0.05
filterCompacted 无 每轮 loop 开始 不释放（只是加载优化） 旧消息不发送 $0

为什么 OpenCode 的方案最适合代码 Agent？

代码开发的核心记忆需求是："我改了什么、为什么改、接下来改什么"——
这是高度时序化、因果化的信息。向量检索擅长"找到相关的"，但不擅长"理解过程的"。

OpenCode 的摘要模板完美契合：Goal → Instructions → Discoveries → Accomplished → Files。
它不是在"搜索记忆"，而是在写交接文档——让下一个 Agent（或压缩后的自己）能无缝接手。
