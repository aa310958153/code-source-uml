一、Plugin 系统全景

OpenCode 的 Plugin 系统是一个三层插件架构，支持从认证到代码分析的各种功能扩展。所有插件最终汇聚到同一个 hooks 数组中，通过统一的 trigger() 函数链式调用。

二、三层插件来源

1. Internal Plugins — 内置硬编码

// plugin/index.ts
const INTERNAL_PLUGINS: PluginInstance[] = [
    CodexAuthPlugin,
    CopilotAuthPlugin,
    GitlabAuthPlugin,
]

这三个插件随 OpenCode 源码一起编译，不需要安装，直接 import。它们处理的是各种 AI Provider 的 OAuth 认证流程：

插件功能
CodexAuthPluginOpenAI Codex 认证
CopilotAuthPluginGitHub Copilot 认证（含 GPT-5 判断）
GitlabAuthPluginGitLab AI Gateway 认证

2. Builtin NPM — 默认安装

const BUILTIN = ["opencode-anthropic-auth@0.0.13"]

// 除非 OPENCODE_DISABLE_DEFAULT_PLUGINS 环境变量被设置
if (!Flag.OPENCODE_DISABLE_DEFAULT_PLUGINS) {
    plugins = [...BUILTIN, ...plugins]
}

Anthropic 的 OAuth 认证作为独立 NPM 包发布，默认安装。用户可以通过环境变量 OPENCODE_DISABLE_DEFAULT_PLUGINS 禁用。

3. User Plugins — 用户配置

// opencode.json
{
    "plugin": [
        "my-opencode-plugin@1.0.0",
        "file:///path/to/local-plugin.js"
    ]
}

用户在配置文件的 plugin 字段指定。支持两种格式：

NPM 包：package-name@version — 自动安装到临时目录
本地文件：file:///absolute/path.js — 直接 import

三、初始化流程详解

Plugin 初始化在 state() 中完成，通过 Instance.state() 绑定到项目实例的生命周期：

const state = Instance.state(async () => {
    // 1. 创建 SDK client（插件通过它与 OpenCode API 交互）
    const client = createOpencodeClient({
        baseUrl: "http://localhost:4096",
        directory: Instance.directory,
        fetch: async (...args) => Server.App().fetch(...args),
    })

    // 2. 构建 PluginInput（注入给每个插件的上下文）
    const input: PluginInput = {
        client,          // OpenCode SDK 客户端
        project: Instance.project,
        worktree: Instance.worktree,
        directory: Instance.directory,
        serverUrl: Server.url(),
        $: Bun.$,        // Bun shell
    }

    // 3. 加载 Internal Plugins
    for (const plugin of INTERNAL_PLUGINS) {
        const init = await plugin(input)
        if (init) hooks.push(init)
    }

    // 4. 加载 Builtin + User Plugins（NPM 包或本地文件）
    for (let plugin of plugins) {
        if (!plugin.startsWith("file://")) {
            plugin = await BunProc.install(pkg, version)
        }
        await import(plugin).then(async (mod) => {
            const seen = new Set()
            for (const [name, fn] of Object.entries(mod)) {
                if (seen.has(fn)) continue  // 去重
                seen.add(fn)
                hooks.push(await fn(input))
            }
        })
    }

    return { hooks, input }
})
SDK Client 的特殊设计：插件获得的 client 并非真正发 HTTP 请求，而是通过 Server.App().fetch(...args) 直接在进程内调用 Hono 路由。这意味着插件的 API 调用零网络开销。

四、Hook 触发机制

所有插件注册的 hooks 汇聚在一个数组中，通过 trigger() 函数链式调用：

export async function trigger<
    Name extends Exclude<keyof Required<Hooks>, "auth" | "event" | "tool">,
    Input = Parameters<Required<Hooks>[Name]>[0],
    Output = Parameters<Required<Hooks>[Name]>[1],
>(name: Name, input: Input, output: Output): Promise<Output> {
    if (!name) return output
    for (const hook of await state().then((x) => x.hooks)) {
        const fn = hook[name]
        if (!fn) continue
        await fn(input, output)  // 每个 hook 可以修改 output
    }
    return output
}

调用链模式

// 调用示例：在 LLM 请求前修改 prompt
const result = await Plugin.trigger(
    "session.prompt",           // hook 名称
    { sessionID: "abc" },       // input（只读上下文）
    { messages: [...], ... }    // output（可修改的数据）
)
// 每个插件的 session.prompt hook 都能修改 result

这是一个典型的管道模式：output 对象在多个插件之间传递，每个插件可以读取和修改它。类型系统保证了 input/output 的类型安全。

注意排除的 Hook 类型：auth、event、tool 这三个 hook 不通过 trigger() 调用。auth 由 Provider 直接调用，event 由 Bus.subscribeAll 分发，tool 在工具注册时直接集成。

五、PluginInput 注入上下文

每个插件在初始化时收到一个 PluginInput 对象，包含与 OpenCode 交互所需的全部上下文：

字段类型说明
clientOpencodeClientSDK 客户端，可调用 Session/Message/Config 等 API
projectProject当前项目信息（ID、名称等）
worktreestringGit worktree 根目录
directorystring项目工作目录
serverUrlstringOpenCode Server 的 URL

// 插件示例：初始化函数接收 input，返回 hooks
export default function myPlugin(input: PluginInput): Hooks {
    return {
        // 在 config 加载后修改配置
        config(cfg) {
            cfg.instructions?.push("Always use TypeScript")
        },
        // 监听所有 Bus 事件
        event({ event }) {
            if (event.type === "session.updated") {
                console.log("Session updated:", event.properties.id)
            }
        },
    }
}

六、内置插件解析

三个 Internal Plugins 各自处理一个 AI Provider 的认证流程。它们在 Plugin 系统最早期加载：

// 加载顺序
// 1. Internal Plugins — 直接导入，零安装
for (const plugin of INTERNAL_PLUGINS) {
    log.info("loading internal plugin", { name: plugin.name })
    const init = await plugin(input).catch((err) => {
        log.error("failed to load internal plugin", { name: plugin.name, error: err })
    })
    if (init) hooks.push(init)
}
容错设计：即使某个内置插件加载失败，也不会阻断其他插件。错误被 catch 后只记录日志，不会 throw。这对于认证类插件尤为重要 — 用户可能只使用其中一个 Provider。

七、NPM 包插件安装与加载

对于非 file:// 前缀的插件，系统会通过 BunProc 自动安装：

for (let plugin of plugins) {
    // 忽略已废弃的旧版插件
    if (plugin.includes("opencode-openai-codex-auth")) continue
    if (plugin.includes("opencode-copilot-auth")) continue

    if (!plugin.startsWith("file://")) {
        // 解析包名和版本
        const lastAtIndex = plugin.lastIndexOf("@")
        const pkg = lastAtIndex > 0 ? plugin.substring(0, lastAtIndex) : plugin
        const version = lastAtIndex > 0 ? plugin.substring(lastAtIndex + 1) : "latest"

        // 安装 NPM 包
        plugin = await BunProc.install(pkg, version).catch((err) => {
            Bus.publish(Session.Event.Error, {
                error: new NamedError.Unknown({
                    message: `Failed to install plugin ${pkg}@${version}`,
                }).toObject(),
            })
            return ""
        })
        if (!plugin) continue
    }

    // 动态 import 并初始化
    await import(plugin).then(async (mod) => {
        const seen = new Set()
        for (const [name, fn] of Object.entries(mod)) {
            if (seen.has(fn)) continue
            seen.add(fn)
            hooks.push(await fn(input))
        }
    })
}

模块去重机制

当一个 NPM 模块同时有 named export 和 default export 指向同一个函数时：

// 例如插件模块：
export const myPlugin = (input) => ({ config(cfg) { ... } })
export default myPlugin  // 同一个引用

// 没有去重会初始化两次！
// seen Set 通过引用比较避免重复初始化
const seen = new Set<PluginInstance>()
for (const [name, fn] of Object.entries(mod)) {
    if (seen.has(fn)) continue  // 跳过重复引用
    seen.add(fn)
    hooks.push(await fn(input))
}

八、本地文件插件

以 file:// 开头的插件直接从本地路径加载，无需安装：

// opencode.json
{
    "plugin": [
        "file:///Users/dev/.opencode/plugins/my-auth.js"
    ]
}

// 加载时跳过 BunProc.install，直接 import
if (!plugin.startsWith("file://")) {
    plugin = await BunProc.install(pkg, version)
}
await import(plugin)  // file:// URL 可被 Bun/Node 直接 import

此外，.opencode/plugins/ 目录下的 .ts/.js 文件也会被自动发现并作为本地插件加载：

// config.ts — loadPlugin()
async function loadPlugin(dir: string) {
    const plugins: string[] = []
    for (const item of await Glob.scan("{plugin,plugins}/*.{ts,js}", {
        cwd: dir,
        absolute: true,
    })) {
        plugins.push(pathToFileURL(item).href)  // 转为 file:// URL
    }
    return plugins
}
本地插件开发很方便：放在 .opencode/plugins/ 下，无需修改 opencode.json，系统自动扫描并转为 file:// URL 加载。

九、事件分发机制

Plugin.init() 除了调用 config hook 外，还会订阅 Bus 的所有事件并分发给插件：

export async function init() {
    const hooks = await state().then((x) => x.hooks)
    const config = await Config.get()

    // 1. 调用每个插件的 config hook（可修改配置）
    for (const hook of hooks) {
        await hook.config?.(config)
    }

    // 2. 订阅所有 Bus 事件，转发给插件的 event hook
    Bus.subscribeAll(async (input) => {
        const hooks = await state().then((x) => x.hooks)
        for (const hook of hooks) {
            hook["event"]?.({ event: input })
        }
    })
}

十、去重与优先级策略

插件可能从多个配置源重复声明，系统通过 deduplicatePlugins() 确保每个插件只加载一次：

export function deduplicatePlugins(plugins: string[]): string[] {
    const seenNames = new Set<string>()
    const uniqueSpecifiers: string[] = []

    // 反向遍历：后出现的（高优先级）先被记录
    for (const specifier of plugins.toReversed()) {
        const name = getPluginName(specifier)
        if (!seenNames.has(name)) {
            seenNames.add(name)
            uniqueSpecifiers.push(specifier)
        }
    }

    // 再反转恢复原始顺序
    return uniqueSpecifiers.toReversed()
}

// getPluginName 提取规范化名称
getPluginName("file:///path/to/foo.js")  // "foo"
getPluginName("oh-my-opencode@2.4.3")    // "oh-my-opencode"
getPluginName("@scope/pkg@1.0.0")        // "@scope/pkg"

优先级顺序（从高到低）

优先级来源说明
1（最高）.opencode/plugins/ 目录本地插件目录
2项目 opencode.json项目级配置
3全局 plugin/ 目录用户级插件
4全局 opencode.json用户级配置

同名插件取高优先级版本，低优先级版本被丢弃。

十一、错误处理与容错

Plugin 系统在多个层面实现了容错：

// 1. Internal Plugin 加载失败 → 记录日志，继续
for (const plugin of INTERNAL_PLUGINS) {
    const init = await plugin(input).catch((err) => {
        log.error("failed to load internal plugin", { name: plugin.name, error: err })
    })
    if (init) hooks.push(init)  // 失败则 init 为 undefined
}

// 2. NPM 安装失败 → 发 Error 事件，跳过
plugin = await BunProc.install(pkg, version).catch((err) => {
    Bus.publish(Session.Event.Error, {
        error: new NamedError.Unknown({
            message: `Failed to install plugin ${pkg}@${version}`,
        }).toObject(),
    })
    return ""
})

// 3. 插件初始化失败 → 发 Error 事件，跳过
await import(plugin).catch((err) => {
    Bus.publish(Session.Event.Error, {
        error: new NamedError.Unknown({
            message: `Failed to load plugin ${plugin}`,
        }).toObject(),
    })
})
关键设计决策：任何插件的加载失败都不会阻止整个系统启动。错误通过 Bus.publish 发布为 Session.Event.Error，前端可以在 UI 上展示这些错误。这是为了确保即使第三方插件 NPM 服务器不可达，用户仍然可以正常使用 OpenCode。
