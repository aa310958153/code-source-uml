一、核心洞察：Guides 层是什么

一句话定义：Guides 层是 Agent 的"大脑操作系统" —— 它不参与推理，但决定了 Agent 以什么身份、遵循什么规则、拥有什么信息来开始每一次推理。

如果把 LLM 比作大脑，那 Guides 就是注入大脑的操作系统：身份认同、行为准则、环境认知。

二、System Prompt 的三层组装

每一次 LLM 调用，System Prompt 都由以下三层动态拼装而成：

// llm.ts — System Prompt 组装核心逻辑
const system = []
system.push(
  [
    // 1. 使用 agent prompt 或 provider prompt
    ...(input.agent.prompt ? [input.agent.prompt]
       : isCodex ? [] : SystemPrompt.provider(input.model)),
    // 2. 外部传入的 system（environment + instructions）
    ...input.system,
    // 3. 用户消息自带的 system
    ...(input.user.system ? [input.user.system] : []),
  ]
  .filter((x) => x)
  .join("\n"),
)

三、Provider Prompt：模型人格选择

不同的 LLM 模型需要不同的引导风格。SystemPrompt.provider() 根据模型 ID 自动选择：

模型Prompt 文件行数核心特点
Claudeanthropic.txt106"You are OpenCode" 标准人格，含 TodoWrite 示例
GPT-5codex_header.txt80精简人格，强调编辑约束和前端任务指南
GPT-4/o1/o3beast.txt148"自主 Agent" 模式，强制 webfetch + 无限迭代
Geminigemini.txt155Gemini 特化版本
Trinitytrinity.txt97Trinity 模型专用
其他qwen.txt109不含 TodoWrite 指令的通用版本
// system.ts — Provider Prompt 选择
export function provider(model: Provider.Model) {
  if (model.api.id.includes("gpt-5")) return [PROMPT_CODEX]
  if (model.api.id.includes("gpt-") || model.api.id.includes("o1"))
    return [PROMPT_BEAST]
  if (model.api.id.includes("gemini-")) return [PROMPT_GEMINI]
  if (model.api.id.includes("claude")) return [PROMPT_ANTHROPIC]
  if (model.api.id.toLowerCase().includes("trinity")) return [PROMPT_TRINITY]
  return [PROMPT_ANTHROPIC_WITHOUT_TODO]
}
为什么不同模型需要不同 Prompt？
不同模型对指令的敏感度不同。Claude 擅长遵循结构化指令，GPT 系列更适合"自主 Agent"式的长迭代， Gemini 对系统提示的解析方式又有差异。Provider Prompt 是模型适配的第一道关卡。

四、Environment：动态环境注入

SystemPrompt.environment() 在每次对话时动态生成，确保 Agent 始终知道当前的运行环境：

// system.ts — 环境信息注入
export async function environment(model: Provider.Model) {
  const project = Instance.project
  return [
    [
      `You are powered by the model named ${model.api.id}.`,
      `Here is some useful information about the environment:`,
      `<env>`,
      `  Working directory: ${Instance.directory}`,
      `  Is directory a git repo: ${project.vcs === "git" ? "yes" : "no"}`,
      `  Platform: ${process.platform}`,
      `  Today's date: ${new Date().toDateString()}`,
      `</env>`,
    ].join("\n"),
  ]
}
为什么要注入环境信息？
LLM 不知道自己在哪台机器上、什么时间、什么目录。这些信息看似简单，但决定了 Agent 能否正确理解文件路径、 判断 Git 操作是否安全、以及理解"今天"是什么时候。

五、Instruction 指令系统：AGENTS.md 发现与加载

5.1 扫描优先级

InstructionPrompt.systemPaths() 按以下优先级扫描指令文件：

1
项目级指令（最高优先级）
从 Instance.directory 向上搜索到 worktree，找到第一个匹配的指令文件
const FILES = ["AGENTS.md", "CLAUDE.md", "CONTEXT.md"] // CONTEXT.md 已弃用
2
全局指令
检查 OPENCODE_CONFIG_DIR/AGENTS.md、~/.config/opencode/AGENTS.md、~/.claude/CLAUDE.md
3
config.instructions（用户配置）
支持文件路径（绝对/相对/~）和 HTTP(S) URL 两种形式
// instruction.ts — 项目级指令发现
export async function systemPaths() {
  const paths = new Set()

  // 1. 项目级：向上查找 AGENTS.md / CLAUDE.md / CONTEXT.md
  for (const file of FILES) {
    const matches = await Filesystem.findUp(file, Instance.directory, Instance.worktree)
    if (matches.length > 0) {
      matches.forEach((p) => paths.add(path.resolve(p)))
      break  // 找到一组就停！不会同时加载 AGENTS.md 和 CLAUDE.md
    }
  }

  // 2. 全局：~/.config/opencode/AGENTS.md 等
  for (const file of globalFiles()) {
    if (await Filesystem.exists(file)) {
      paths.add(path.resolve(file))
      break  // 同样只取第一个存在的
    }
  }

  // 3. config.instructions 文件和 URL
  if (config.instructions) { ... }

  return paths
}
关键设计：break 的语义。 FILES = ["AGENTS.md", "CLAUDE.md", "CONTEXT.md"] 按优先级排列。 找到 AGENTS.md 后直接 break，不会再查找 CLAUDE.md。这避免了同一项目加载多份冲突指令。

六、子目录指令发现（resolve）

当 Agent 通过 Read 工具读取文件时，系统会自动检查该文件所在目录及其父目录是否有指令文件：

// instruction.ts — 子目录指令发现
export async function resolve(messages, filepath, messageID) {
  const system = await systemPaths()   // 已在 system prompt 中的
  const already = loaded(messages)     // 已通过 Read 加载过的
  const results = []

  let current = path.dirname(filepath)
  const root = path.resolve(Instance.directory)

  // 从文件所在目录向上遍历到项目根
  while (current.startsWith(root) && current !== root) {
    const found = await find(current)  // 查找 AGENTS.md 等

    if (found
        && found !== target           // 不是正在读的文件本身
        && !system.has(found)         // 不在 system prompt 中（避免重复）
        && !already.has(found)        // 没有通过 Read 加载过
        && !isClaimed(messageID, found)) {  // Claim 机制防重复
      claim(messageID, found)
      results.push({ filepath: found, content: "Instructions from: " + found + "\n" + ... })
    }
    current = path.dirname(current)
  }
  return results
}
为什么需要子目录指令？
大型项目中，不同子目录可能有不同的编码规范。例如 src/components/AGENTS.md 可以定义 React 组件的编写规范， 而 src/api/AGENTS.md 定义 API 路由的编写规范。Agent 读取哪个目录的文件，就自动获得该目录的规范。

七、Claim 机制：防止重复注入

当同一个消息轮次中 Agent 读取多个文件时，子目录指令可能被多次发现。Claim 机制确保每个指令文件只注入一次：

// instruction.ts — Claim 机制
const state = Instance.state(() => {
  return { claims: new Map() }  // messageID → Set
})

function isClaimed(messageID, filepath) {
  return state().claims.get(messageID)?.has(filepath) ?? false
}

function claim(messageID, filepath) {
  let claimed = state().claims.get(messageID)
  if (!claimed) {
    claimed = new Set()
    state().claims.set(messageID, claimed)
  }
  claimed.add(filepath)
}

// 每轮对话结束时清理
export function clear(messageID) {
  state().claims.delete(messageID)
}
Claim 的生命周期：
• claim() 在 resolve() 中调用，标记"此消息轮次已注入此指令文件"
• clear() 在 loop() 的 using _ = defer(() => clear(messageID)) 中调用
• 一轮对话结束后自动清理，下一轮可以重新注入

八、Config Instructions：用户自定义指令

用户可以在 opencode.json 中配置额外的指令来源：

// opencode.json
{
  "instructions": [
    "./docs/coding-standards.md",       // 相对路径
    "~/global-rules.md",                // Home 目录
    "/absolute/path/rules.md",          // 绝对路径
    "https://example.com/rules.txt"     // HTTP URL
  ]
}

8.1 文件路径解析

// instruction.ts — 文件路径解析
for (let instruction of config.instructions) {
  if (instruction.startsWith("https://")) continue  // URL 单独处理

  if (instruction.startsWith("~/")) {
    instruction = path.join(os.homedir(), instruction.slice(2))
  }

  // 绝对路径：直接 glob 扫描
  // 相对路径：从项目目录向上查找
  const matches = path.isAbsolute(instruction)
    ? await Glob.scan(path.basename(instruction), { cwd: path.dirname(instruction) })
    : await resolveRelative(instruction)  // Filesystem.globUp
}

8.2 URL 远程加载

// instruction.ts — URL 远程指令
const fetches = urls.map((url) =>
  fetch(url, { signal: AbortSignal.timeout(5000) })  // 5秒超时
    .then((res) => res.ok ? res.text() : "")
    .catch(() => "")
    .then((x) => x ? "Instructions from: " + url + "\n" + x : "")
)
URL 指令的价值：团队可以将编码规范托管在统一的服务器上，所有开发者的 OpenCode 自动拉取最新版本。 5 秒超时确保不会因为网络问题阻塞 Agent 启动。

九、Agent Prompt：每个 Agent 的个性化引导

每个 Agent 可以有自己的 prompt 字段。当 Agent 有 prompt 时，它替代 Provider Prompt 成为第一层：

Agent有 Prompt？Prompt 内容设计意图
build无使用 Provider Prompt（anthropic/beast/...）通用执行，依赖模型默认人格
explore✅explore.txt — "文件搜索专家"，只读工具指南覆盖模型人格，专注搜索
plan无使用 Provider Prompt + Plan Reminder保持模型人格，通过 Reminder 约束行为
compaction✅compaction.txt — "对话摘要助手"完全不同的角色定义
title✅title.txt — "标题生成器"极简角色，只需生成标题
summary✅summary.txt — "变更总结器"专注代码变更总结
agent.prompt 覆盖 provider prompt 的逻辑：
agent.prompt ? [agent.prompt] : SystemPrompt.provider(model)

这是一个"全有或全无"的替代关系 —— Agent prompt 不是追加在 Provider prompt 后面，而是完全替代。 因此 Agent prompt 必须包含完整的角色定义和行为指南。

十、Reminders：运行时上下文提醒

Reminders 是在消息处理阶段动态注入的 <system-reminder> 标签，不属于 System Prompt，而是追加在用户消息末尾：

10.1 Plan 模式提醒

// prompt.ts — insertReminders
if (input.agent.name === "plan") {
  userMessage.parts.push({
    type: "text",
    text: PROMPT_PLAN,  // plan.txt: "STRICTLY FORBIDDEN: ANY file edits"
    synthetic: true,
  })
}

10.2 模式切换提醒

// 从 Plan 切换到 Build 时
if (wasPlan && input.agent.name === "build") {
  userMessage.parts.push({
    type: "text",
    text: BUILD_SWITCH,  // "Your operational mode has changed from plan to build"
    synthetic: true,
  })
}

10.3 Max Steps 提醒

// 达到最大步数时，注入为 assistant 消息（不是 system reminder）
messages: [
  ...toModelMessages(msgs, model),
  ...(isLastStep ? [{ role: "assistant", content: MAX_STEPS }] : []),
]
// MAX_STEPS = "CRITICAL - MAXIMUM STEPS REACHED... Do NOT make any tool calls"
为什么 Reminders 不是 System Prompt？
System Prompt 在 API 层面是固定的，部分模型对 System Prompt 中的"禁止"指令遵循度不够。 将 Plan 模式的严格约束放在用户消息中（作为 system-reminder 标签），可以提高模型的遵循度。

十一、LLM.stream 最终组装

所有 Guides 最终在 LLM.stream() 中完成组装。这是整个引导层的终点：

// llm.ts — 缓存优化逻辑
const header = system[0]
await Plugin.trigger("experimental.chat.system.transform", ..., { system })

// 如果 header 未被插件修改，拆分为两段以利用 prompt caching
if (system.length > 2 && system[0] === header) {
  const rest = system.slice(1)
  system.length = 0
  system.push(header, rest.join("\n"))
  // 现在 system = [不变的header, 动态的rest]
  // LLM Provider 可以缓存 header 部分，减少重复计算
}
Prompt Caching 优化：System Prompt 的第一段（Agent Prompt + Environment）在多轮对话中通常不变。 将其单独作为 system[0]，LLM Provider（如 Anthropic）可以缓存这部分的 KV-cache， 后续请求只需传输变化的部分，显著降低延迟和成本。

十二、设计优势总结

🎭

模型适配层

Provider Prompt 为每个 LLM 家族定制人格，最大化各模型的指令遵循能力。Claude 走结构化路线，GPT 走自主 Agent 路线。

📂

分层指令发现

全局 → 项目 → 子目录，三级指令自动发现。大项目不同目录可以有不同规范，Agent 读取文件时自动获得相关规范。

🔒

Claim 防重复

每轮对话中，每个指令文件只注入一次。即使 Agent 读取同一目录的 10 个文件，子目录 AGENTS.md 也只出现一次。

⚡

Prompt Caching

System Prompt 拆分为 header + rest 两段。header 不变时可被 LLM Provider 缓存，降低后续请求的 token 消耗和延迟。

🌐

远程指令支持

config.instructions 支持 HTTP URL，团队可将编码规范托管在统一服务器。5 秒超时确保不会阻塞启动。

🔌

Plugin 扩展点

experimental.chat.system.transform 允许插件在发送前修改 System Prompt。如注入自定义 header、添加项目元数据等。

🧱

Agent 覆盖机制

agent.prompt 全量替代 provider prompt，让特殊用途 Agent（explore/compaction/title）拥有完全不同的角色定义。

📌

运行时 Reminders

Plan/Build 模式切换、Max Steps 限制等运行时状态通过 synthetic text part 注入，比 System Prompt 更灵活、遵循度更高。

🧹

优雅降级

URL 超时返回空串、文件不存在跳过、AGENTS.md 不存在不报错。任何指令源失败都不影响 Agent 正常运行。

一句话总结：OpenCode 的 Guides 层是一个"静态声明 + 动态组装 + 运行时注入"的三层引导架构。
它用 Provider Prompt 适配不同模型、用 AGENTS.md 体系管理项目规范、用 Agent Prompt 定义角色差异、 用 Reminders 处理运行时状态，最终在 LLM.stream() 中完成统一组装并利用 Prompt Caching 优化性能。
整个设计让"告诉 Agent 该怎么做"这件事变得声明式、可组合、可扩展。
