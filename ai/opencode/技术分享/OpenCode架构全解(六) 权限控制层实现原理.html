一、核心洞察：为什么 Agent 需要权限系统

问题的本质：LLM Agent 与普通程序的根本区别 —— 你无法预知它下一步要做什么。
传统程序的行为是确定性的，而 Agent 的行为由模型推理产生——它可能删文件、读 .env、执行任意 Bash 命令、甚至陷入死循环。
权限系统必须回答一个核心问题：这个 Agent 在这个时刻，对这个资源，能做什么？

二、三种动作：allow / deny / ask

权限系统的输出只有三种可能。这是一个极简但完备的设计：

动作含义行为使用场景
allow 自动放行 工具直接执行，用户无感知 read、grep、glob 等只读工具
deny 自动拒绝 抛出 DeniedError，将规则集返回给 LLM plan 模式禁止编辑、子 Agent 禁用工具
ask 请求人工确认 发布事件阻塞等待，用户选择 once/always/reject edit、write、bash 等危险操作
关键设计决策：默认动作是 ask 而非 deny。
这意味着没有配置任何规则时，系统会请求用户确认，而不是静默拒绝。
这保证了 Agent 不会因为"忘记配权限"而卡死。 
// evaluate() 的默认返回值
return match ?? { action: "ask", permission, pattern: "*" }

三、规则评估算法：findLast + Wildcard 匹配

3.1 Rule 数据结构

interface Rule {
  permission: string   // 权限名，如 "edit"、"bash"、"read"
  pattern: string      // 匹配模式，如 "*.env"、"git *"、"*"
  action: "allow" | "deny" | "ask"
}

type Ruleset = Rule[]   // 有序数组，顺序即优先级

3.2 evaluate() 核心算法

评估函数的逻辑非常精炼：将所有 Ruleset 合并成一个平面数组，从后往前找第一个匹配的规则。

function evaluate(
  permission: string,    // 当前请求的权限名
  pattern: string,       // 当前请求的具体模式（如文件路径）
  ...rulesets: Ruleset[] // 多层规则集
): Rule {
  const merged = merge(...rulesets) // 简单 flat() 合并

  const match = merged.findLast(
    (rule) => Wildcard.match(permission, rule.permission)
           && Wildcard.match(pattern, rule.pattern),
  )

  return match ?? { action: "ask", permission, pattern: "*" }
}

3.3 为什么是 findLast？

因为 Ruleset 是一个有序数组，后面的规则覆盖前面的。这是权限系统最核心的设计：

越后追加的规则优先级越高。合并时 defaults → agent → user，用户的配置永远在最末尾，因此拥有最高优先级。

3.4 Wildcard 通配符匹配

function match(str: string, pattern: string) {
  // 统一路径分隔符
  str = str.replaceAll("\\", "/")
  pattern = pattern.replaceAll("\\", "/")

  // 转义正则特殊字符，* → .*, ? → .
  let escaped = pattern
    .replace(/[.+^${}()|[\]\\]/g, "\\$&")
    .replace(/\*/g, ".*")
    .replace(/\?/g, ".")

  // "ls *" 同时匹配 "ls" 和 "ls -la"
  if (escaped.endsWith(" .*"))
    escaped = escaped.slice(0, -3) + "( .*)?"

  return new RegExp("^" + escaped + "$", flags).test(str)
}
匹配维度示例说明
permission 名匹配"*" 匹配所有权限用于全局 allow/deny
pattern 路径匹配"*.env" 匹配所有 .env 文件用于细粒度文件控制

四、三层规则合并：defaults → agent → user

每个 Agent 的最终权限是三层 Ruleset 合并的结果：

merge() 极简实现

function merge(...rulesets: Ruleset[]): Ruleset {
  return rulesets.flat() // 就是数组拼接！顺序即优先级
}
设计之美：merge 就是 flat()。没有复杂的逻辑合并、没有冲突检测 —— 完全依赖数组顺序。
这让权限系统极其可预测：你只需要知道规则在数组中的位置，就能判断它是否生效。

五、内置 Agent 权限矩阵

OpenCode 内置 7 个 Agent，每个有精心设计的权限策略：

Agent模式核心权限策略设计意图
buildprimarydefaults + question/plan_enter: allow默认 Agent，全功能执行
planprimaryedit: deny * + allow .opencode/plans/*.md规划模式，只允许写计划文件
exploresubagent*: deny + grep/glob/read/bash: allow代码探索，只读不写
generalsubagentdefaults + todo: deny通用任务，不允许操作 TODO
compactionprimary(hidden)*: deny压缩对话，纯 LLM 调用无工具
titleprimary(hidden)*: deny生成标题，纯 LLM 调用无工具
summaryprimary(hidden)*: deny生成摘要，纯 LLM 调用无工具
explore Agent 的巧妙设计：先 *: deny 全局禁止，再逐个 allow 只读工具。
这是"最小权限原则"的经典实践 —— 默认拒绝一切，只开放必要能力。

六、工具层权限请求流程（ctx.ask）

每个工具在执行敏感操作前，通过 ctx.ask() 向权限系统发起请求：

// edit.ts — 编辑文件前请求权限
await ctx.ask({
  permission: "edit",                              // 请求 "edit" 权限
  patterns: [path.relative(worktree, filePath)],    // 具体文件路径
  always: ["*"],                                   // "always"时记住什么
  metadata: { filepath, diff },                    // 展示给用户看的上下文
})

// read.ts — 读取文件前请求权限
await ctx.ask({
  permission: "read",
  patterns: [filepath],
  always: ["*"],
  metadata: {},
})

// bash.ts — 执行命令前请求权限（使用 BashArity 解析出的模式）
await ctx.ask({
  permission: "bash",
  patterns: [command],                    // 如 "git status"
  always: [head + " *", head],           // 如 "git *" 和 "git"
  metadata: { command },
})

ctx.ask 内部做了什么？

1
合并 Agent 权限 + 已批准规则
const ruleset = PermissionNext.merge(agent.permission, session.permission)
2
对每个 pattern 调用 evaluate()
根据合并后的 ruleset 判断该操作是 allow / deny / ask
3
根据结果执行或阻塞
allow → continue | deny → throw DeniedError | ask → Promise 阻塞等用户回复
async function ask(input) {
  for (const pattern of input.patterns ?? []) {
    const rule = evaluate(input.permission, pattern, ruleset, approved)

    if (rule.action === "deny")
      throw new DeniedError(...)    // 直接抛错，工具中止

    if (rule.action === "ask")
      return new Promise((resolve, reject) => {
        pending[id] = { info, resolve, reject }
        Bus.publish(Event.Asked, info) // 发布事件，UI 弹出确认框
      })                              // Promise 阻塞，直到用户回复

    if (rule.action === "allow")
      continue                        // 放行，检查下一个 pattern
  }
}

七、会话级动态授权（once / always / reject）

当 evaluate 返回 ask 时，系统会阻塞等待用户回复。用户有三种选择：

always 的级联效应

当用户选择 always 时，不仅当前请求被放行，系统还会自动扫描同一 session 的所有 pending 请求：

if (input.reply === "always") {
  // 1. 将 "always" 模式追加到 approved 列表
  for (const pattern of info.always)
    approved.push({ permission, pattern, action: "allow" })

  // 2. 当前请求放行
  existing.resolve()

  // 3. 级联：检查同 session 所有 pending 请求
  for (const [id, pending] of Object.entries(s.pending)) {
    if (pending.info.sessionID !== sessionID) continue
    // 用更新后的 approved 重新评估
    const ok = pending.info.patterns.every(
      (p) => evaluate(pending.info.permission, p, approved).action === "allow"
    )
    if (!ok) continue
    // 新规则覆盖了 → 自动放行！
    delete s.pending[id]
    pending.resolve()
  }
}
实际场景：Agent 同时请求 5 个文件的 edit 权限 → 用户对第一个选择 "always" → 其余 4 个自动放行。
因为 always 追加的规则 "edit": "*" 覆盖了所有 pending 的文件模式匹配。

reject 的级联效应

// reject 会一次性拒绝同 session 所有 pending 请求
if (input.reply === "reject") {
  existing.reject(...)
  for (const [id, pending] of Object.entries(s.pending)) {
    if (pending.info.sessionID === sessionID) {
      delete s.pending[id]
      pending.reject(new RejectedError())
    }
  }
}

这个设计防止了"用户拒绝了一个操作，但 Agent 的后续操作还在等待确认"的尴尬情况。

八、Bash 命令的特殊权限处理

Bash 是最危险的权限维度 —— 一条命令可以做任何事。OpenCode 对 Bash 有双重权限控制：

8.1 外部目录检测

Bash 命令中的文件路径会被 tree-sitter 解析，如果路径在项目外，触发 external_directory 权限：

// bash.ts — 文件操作命令的路径检查
const FILE_OP_COMMANDS = ["cat", "rm", "mv", "cp", "chmod", ...]

if (FILE_OP_COMMANDS.includes(cmd)) {
  for (const arg of args) {
    const resolved = realpath(arg)           // 解析真实路径
    if (!Instance.containsPath(resolved))     // 是否在项目内？
      await ctx.ask({ permission: "external_directory", patterns: [resolved] })
  }
}

8.2 BashArity 命令粒度授权

不同命令有不同的"粒度"，由 BashArity 字典定义：

命令前缀arityalways 时记住的模式含义
git2git * 和 git记住"git 的所有命令"
npm run3npm run * 和 npm run记住"npm run 的所有脚本"
docker compose3docker compose *记住"docker compose 的所有命令"
ls1ls只记住"ls 本身"
// bash.ts 中 arity 的使用
const prefix = BashArity.prefix(command)
// 如 "git status" → prefix = ["git *", "git"]
// 如 "npm run dev" → prefix = ["npm run *", "npm run"]

await ctx.ask({
  permission: "bash",
  patterns: [command],        // ["git status"]
  always: prefix,             // ["git *", "git"] → 用户选 always 后记住整个 git 命令族
  metadata: { command },
})

九、外部目录保护机制

Agent 操作项目目录外的文件时，触发 external_directory 权限。这是一个独立的权限维度：

// defaults 中 external_directory 的默认配置
external_directory: {
  "*": "ask",                          // 所有外部目录都需要确认
  ...whitelistedDirs.map(dir => [dir, "allow"]), // 技能目录自动放行
}
检查点触发条件保护对象
工具层 assertExternalDirectory()Read/Edit/Write/Glob/Grep 等文件工具通过文件路径检测
Bash 层 tree-sitter 解析Bash 中的文件操作命令（cat/rm/mv/cp 等）通过 AST 提取路径参数
Truncate 输出文件的特殊处理：被截断的工具输出保存在 .opencode/truncation/ 目录下。
这个目录位于项目外，但系统自动将其加入白名单，确保 Agent 可以读取截断文件而不触发权限请求。

十、disabled() 预过滤与工具裁剪

在 Agent 启动时，权限系统会预扫描 Ruleset，找出被全局 deny 的工具，直接从工具列表中移除：

function disabled(tools: string[], ruleset: Ruleset): Set {
  const result = new Set()
  for (const tool of tools) {
    // write/edit/patch/multiedit 都映射到 "edit" 权限
    const permission = EDIT_TOOLS.includes(tool) ? "edit" : tool

    const rule = ruleset.findLast(
      (r) => Wildcard.match(permission, r.permission)
    )
    if (!rule) continue
    // pattern="*" + action="deny" → 全局禁止
    if (rule.pattern === "*" && rule.action === "deny")
      result.add(tool)
  }
  return result
}
为什么需要预过滤？
如果 LLM 看到一个工具但永远无法使用它，会浪费 token 去尝试调用，然后被 deny。
disabled() 在工具注册阶段就移除这些工具，LLM 根本不知道它们的存在。

这个函数只检查"全局 deny"（pattern="*" 且 action="deny"）， 不处理细粒度的 deny 规则（如 *.env: deny），因为那些需要在运行时根据具体文件路径判断。

十二、设计优势总结

🧩

声明式配置

权限是数据而非代码。用户通过 JSON 配置，不需要写 TypeScript。

"permission": {
  "bash": { "npm *": "allow" },
  "webfetch": "deny"
}
📐

极简合并模型

merge = flat()。没有复杂的冲突检测或优先级计算。数组位置决定一切。defaults → agent → user，后者覆盖前者。

🔍

Wildcard 双维匹配

同时匹配 permission 名和 pattern 路径。一条规则可以同时控制多个工具的多个文件。"*": "deny" 禁止所有。

💬

渐进式信任

ask → once（临时信任）→ always（永久信任）。用户可以逐步建立对 Agent 的信任，而不是一开始就配置所有权限。

🧱

Agent 隔离

不同 Agent 不同权限。explore 只读，build 全功能，plan 只写计划文件。子 Agent 天然受限，防止越权。

⚡

预过滤优化

disabled() 在工具注册阶段移除被全局 deny 的工具。LLM 不看到不存在的工具，不浪费 token。

🛡️

多层防护

Bash 命令同时检查 bash 权限和 external_directory 权限。双重保险：tree-sitter AST 解析 + realpath 路径检查。

🔄

级联响应

always 自动放行同 session 匹配的 pending 请求。reject 自动拒绝同 session 所有 pending。一个决策，批量生效。

📊

三类错误语义

RejectedError：用户拒绝，停止执行。
CorrectedError：用户纠正，带反馈继续。
DeniedError：规则拒绝，返回规则集给 LLM。

一句话总结：OpenCode 的权限系统是一个"声明式规则 + 运行时评估 + 人机交互式授权"的三层架构。
它用极简的 flat() 合并和 findLast 匹配，解决了传统权限系统中复杂的冲突检测和优先级计算问题。
同时通过 always/reject 级联和 disabled() 预过滤，在安全性和效率之间取得了优雅的平衡。
