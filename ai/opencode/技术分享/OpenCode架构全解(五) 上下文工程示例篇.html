模型决策 #2：Reasoning（推理）

// 模型的 Reasoning 过程
我看到 parseConfig 函数的问题了。在第 11-15 行：

  obj.database.host
  obj.database.port
  obj.database.credentials.username
  obj.database.credentials.password

如果 raw 对象中 database 字段不存在或为 undefined，
那么 obj.database 就是 undefined，
访问 obj.database.host 就会抛 TypeError: Cannot read properties of undefined。

同样的问题存在于 credentials 层——如果 database.credentials 不存在，
访问 .username 也会报错。

修复方案：使用可选链 (?.) 和空值合并 (??)，
或者先检查嵌套对象是否存在。

让我使用 Edit 工具修复这段代码。

上下文增长全景图

Step > 1 时的用户消息包装机制

如果在 Step 2 执行期间，用户发送了新消息 "对了，也帮我加上单元测试"，这条消息在 Step 3 的上下文中会被特殊包装：

// Step 3 中模型看到的用户新消息（被包装后）
{
  role: "user",
  content: [
    {
      type: "text",
      text: "<system-reminder>\nThe user sent the following message:\n对了，也帮我加上单元测试\n\nPlease address this message and continue with your tasks.\n</system-reminder>"
    }
  ]
}
为什么包装？在多轮循环中，如果用户的新消息直接作为普通 user 消息出现，模型可能会将其视为新的独立任务，放弃当前正在执行的工作。包装为 <system-reminder> 后，模型会理解为"这是用户在当前任务进行中的补充说明"，而非新任务。

Processor 内部事件流 — 工具调用的全生命周期

SessionProcessor.process() 通过 stream.fullStream 接收 LLM 的流式输出，每个事件类型对应不同的处理逻辑：

// processor.ts - 流式事件处理顺序（以 Step 2 为例）

// 1. start-step → 记录 git snapshot（用于 diff）
"start-step"  →  snapshot = await Snapshot.track()

// 2. reasoning → 模型的思考过程（thinking tokens）
"reasoning-start" → 创建 ReasoningPart
"reasoning-delta" → 追加思考文本
"reasoning-end"   → 完成 ReasoningPart

// 3. text → 模型输出的文本（给用户看的）
"text-start"  → 创建 TextPart
"text-delta"  → 追加文本 → Session.updatePartDelta() → SSE 推送给前端
"text-end"    → 完成 TextPart → Plugin.trigger("experimental.text.complete")

// 4. tool-call → 模型决定调用工具
"tool-input-start" → 创建 ToolPart (status: "pending")
"tool-call"        → 更新 ToolPart (status: "running") + 检查 Doom Loop
                     → 执行工具 → 权限请求 → 实际执行 → 结果返回
"tool-result"      → 更新 ToolPart (status: "completed") + 记录 output

// 5. finish-step → 统计 token/cost
"finish-step" → 计算 usage → 检查 isOverflow() → 标记 needsCompaction
              → Snapshot.patch() → 记录文件变更

// 6. finish → 整个流结束
"finish" → 清理 → 返回结果给 loop()

Doom Loop 检测

如果最近 3 个工具调用是同一个工具 + 完全相同的参数，触发 doom_loop 权限请求：

// processor.ts - Doom Loop 检测
const DOOM_LOOP_THRESHOLD = 3
const lastThree = parts.slice(-DOOM_LOOP_THRESHOLD)

if (lastThree.length === DOOM_LOOP_THRESHOLD &&
    lastThree.every(p =>
        p.type === "tool" &&
        p.tool === value.toolName &&
        JSON.stringify(p.state.input) === JSON.stringify(value.input)
    )) {
    // 请求 doom_loop 权限 — 让用户确认是否继续
    await PermissionNext.ask({
        permission: "doom_loop",
        patterns: [value.toolName],
        ...
    })
}

这防止了模型陷入"读取文件 → 发现问题 → 再读取同一文件"的无限循环。

核心规律总结

1. 模型永远看不到代码

模型看到的不是"文件"，而是工具返回的格式化文本（带行号的 content、diff 输出、bash stdout）。工具的 output 格式直接决定模型的理解质量。

2. System Prompt 每轮重建

每轮循环都重新调用 environment() + instructions() + provider()，但内容基本不变。Prompt Caching 通过 2 段结构最大化 KV-cache 命中率。

3. 工具集每轮重解析

resolveTools() 每轮都执行完整的管线：注册 → Schema 转换 → Plugin Hook → MCP 合并 → 权限过滤。工具集可能因权限变化而动态增减。

4. 历史消息只增不减

每轮循环增加 assistant 消息（text + tool calls + results）。消息历史线性增长，直到 Compaction 触发压缩，或 Prune 裁剪旧工具输出。

5. 模型的决策基于全量上下文

每次 LLM 调用，模型都看到完整的 System Prompt + 全部历史消息 + 全部工具定义。它的"推理"本质上是在这个上下文空间中的模式匹配。

6. finish_reason 是唯一退出信号

模型通过 finish_reason 告诉系统它的意图：stop = 我完成了，tool_use = 我要调用工具，length = 输出被截断。loop() 据此决定 continue 或 break。
