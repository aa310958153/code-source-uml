第一章 核心洞察：什么是上下文工程

上下文 ≠ 提示词

传统"提示词工程"关注的是写给模型什么内容。而 OpenCode 的"上下文工程"关注的是在每一轮循环迭代中，如何动态组装、过滤、压缩、注入信息，使模型在正确的时刻看到正确的上下文。

这包括：System Prompt 分层组装 → 模型适配模板选择 → 工具集动态解析 → 消息历史格式转换 → Reminders 条件注入 → 上下文溢出检测 → Compaction 压缩与裁剪。

第二章 loop() 核心循环 — 上下文工程的心跳

prompt.ts 的 loop() 函数是整个上下文工程的引擎。它是一个 while(true) 无限循环，每一轮迭代代表一次完整的 LLM 交互。

// prompt.ts - loop() 核心骨架 (简化)
let step = 0
while (true) {
    if (abort.aborted) break
    
    // 1. 消息流过滤 — 获取压缩后的消息历史
    let msgs = await MessageV2.filterCompacted(MessageV2.stream(sessionID))
    
    // 2. 回溯查找关键消息
    //    lastUser: 最近的用户消息
    //    lastAssistant: 最近的助手消息
    //    lastFinished: 最近完成(非tool-calls)的助手消息
    //    tasks: 待处理的 subtask/compaction 任务
    
    // 3. 退出条件判断
    if (lastAssistant?.finish && !["tool-calls","unknown"].includes(lastAssistant.finish)
        && lastUser.id < lastAssistant.id) break
    
    step++
    
    // 4. 处理待执行任务 (subtask → compaction)
    if (task?.type === "subtask") { /* 执行 TaskTool */ continue }
    if (task?.type === "compaction") { /* 执行压缩 */ continue }
    
    // 5. 上下文溢出检测 → 触发 Compaction
    if (lastFinished && await SessionCompaction.isOverflow({tokens, model})) {
        await SessionCompaction.create({sessionID, auto: true})
        continue
    }
    
    // 6. 正常处理流程
    msgs = await insertReminders({messages, agent, session})  // 注入提醒
    const tools = await resolveTools({agent, model, ...})     // 解析工具集
    const system = [...environment(), ...instructions()]       // 构建 System Prompt
    
    // 7. step > 1 时包装用户消息队列
    if (step > 1) { /* 包装为 system-reminder */ }
    
    // 8. 调用 LLM
    const result = await processor.process({
        system, messages: toModelMessages(msgs, model), tools, model
    })
    
    // 9. 结果判断: stop → break | compact → 触发压缩 | continue → 下一轮
}

第三章 System Prompt 三层组装架构

System Prompt 不是静态文本，而是由 llm.ts stream() 函数在每次 LLM 调用时动态组装的三层结构：

// llm.ts - stream() 中 System Prompt 组装
const system = []
system.push(
    [
        // 第一层：Agent Prompt 或 Provider Prompt（二选一）
        ...(input.agent.prompt
            ? [input.agent.prompt]
            : isCodex ? [] : SystemPrompt.provider(input.model)),
        
        // 第二层：Environment + Instructions（外部传入的 system 数组）
        ...input.system,
        
        // 第三层：用户消息级别的自定义 system
        ...(input.user.system ? [input.user.system] : []),
    ]
    .filter((x) => x)
    .join("\n")
)

Prompt Caching 优化

组装完成后，OpenCode 会检测 System Prompt 的 header 部分是否变化。如果 header 未变（仅尾部 Instructions 变化），则将 System 拆分为 [header, rest] 两段，让 LLM Provider 可以缓存 header 的 KV-cache，大幅减少重复计算。

// llm.ts - Prompt Caching 逻辑
const header = system[0]
await Plugin.trigger("experimental.chat.system.transform", {...}, {system})

// 如果 header 未变，重组为 2 段结构
if (system.length > 2 && system[0] === header) {
    const rest = system.slice(1)
    system.length = 0
    system.push(header, rest.join("\n"))
}

第五章 Environment 动态环境注入

SystemPrompt.environment(model) 在每次调用时动态注入运行时环境信息，让模型感知当前上下文。

// system.ts - environment() 函数
export async function environment(model: Provider.Model) {
    const project = Instance.project
    return [
        [
            `You are powered by the model named ${model.api.id}.
             The exact model ID is ${model.providerID}/${model.api.id}`,
            `Here is some useful information about the environment:`,
            `<env>`,
            `  Working directory: ${Instance.directory}`,
            `  Is directory a git repo: ${project.vcs === "git" ? "yes" : "no"}`,
            `  Platform: ${process.platform}`,
            `  Today's date: ${new Date().toDateString()}`,
            `</env>`,
            `<directories>`,
            `  ${/* 项目目录树（当前已禁用） */}`,
            `</directories>`,
        ].join("\n"),
    ]
}
注入的信息：模型名称和精确ID → 工作目录路径 → Git 仓库状态 → 操作系统平台 → 当前日期。这些信息让模型能够根据环境调整行为（如在 Windows 上使用不同的路径格式）。

第六章 Instructions 指令系统 — 用户侧上下文注入

InstructionPrompt.system() 从多个来源加载用户/项目级指令，作为 System Prompt 的第二层注入。

// instruction.ts - 指令来源优先级
const FILES = [
    "AGENTS.md",    // OpenCode 标准配置文件
    "CLAUDE.md",    // Claude Code 兼容配置
    "CONTEXT.md",   // 已弃用
]

// 加载顺序：
// 1. 项目级：从 Instance.directory 向上查找到 worktree
//    → AGENTS.md > CLAUDE.md > CONTEXT.md（找到即停）
// 2. 全局级：
//    → $OPENCODE_CONFIG_DIR/AGENTS.md
//    → ~/.config/opencode/AGENTS.md
//    → ~/.claude/CLAUDE.md（兼容模式）
// 3. 自定义：config.instructions 配置项
//    → 本地路径（支持 ~ 和相对路径，向上查找）
//    → URL（HTTP fetch，5秒超时）

目录级 Instructions 自动发现

当 Read 工具读取文件时，InstructionPrompt.resolve() 会自动查找该文件所在目录及所有父目录中的 AGENTS.md/CLAUDE.md，将其注入为工具输出的附加上下文。这实现了"读什么文件就看到什么规范"的智能上下文关联。

// instruction.ts - resolve() 目录级指令发现
export async function resolve(messages, filepath, messageID) {
    let current = path.dirname(target)
    while (current.startsWith(root) && current !== root) {
        const found = await find(current)  // 查找 AGENTS.md / CLAUDE.md
        if (found && !system.has(found) && !already.has(found)) {
            claim(messageID, found)  // 标记已加载，避免重复
            results.push({filepath: found, content: "Instructions from: " + found + "\n" + content})
        }
        current = path.dirname(current)  // 向上遍历
    }
}

insertReminders() 根据当前 Agent 模式和会话状态，向用户消息中动态注入特殊的 system-reminder 文本块。

你只能观察、分析和规划。任何修改尝试都是严重违规。

7.4 用户消息队列包装

当 step > 1（即循环进入多轮迭代）时，如果用户在 LLM 执行期间发送了新消息，这些消息会被包装为 <system-reminder> 格式，防止模型将其视为新的独立请求：

// prompt.ts - step > 1 时的消息包装
if (step > 1 && lastFinished) {
    for (const msg of msgs) {
        if (msg.info.role !== "user" || msg.info.id <= lastFinished.id) continue
        for (const part of msg.parts) {
            if (part.type !== "text" || part.ignored || part.synthetic) continue
            part.text = [
                "<system-reminder>",
                "The user sent the following message:",
                part.text,
                "",
                "Please address this message and continue with your tasks.",
                "</system-reminder>",
            ].join("\n")
        }
    }
}

第八章 工具集动态解析管线

resolveTools() 在每轮循环中动态构建工具集，经过多阶段过滤和增强：

// prompt.ts - resolveTools() 完整管线
async function resolveTools(input) {
    const tools = {}
    
    // 阶段①：内置 + 自定义 + Plugin 工具（已经模型适配过滤）
    for (const item of await ToolRegistry.tools({modelID, providerID}, agent)) {
        // 阶段②：Schema 转换（适配不同 Provider 的 JSON Schema 方言）
        const schema = ProviderTransform.schema(model, z.toJSONSchema(item.parameters))
        
        // 包装工具执行：添加 Plugin hooks + 权限请求 + 元数据更新
        tools[item.id] = tool({
            description: item.description,
            inputSchema: jsonSchema(schema),
            async execute(args, options) {
                const ctx = context(args, options)
                await Plugin.trigger("tool.execute.before", {...}, {args})
                const result = await item.execute(args, ctx)
                await Plugin.trigger("tool.execute.after", {...}, result)
                return result
            },
        })
    }
    
    // 阶段④：合并 MCP 工具
    for (const [key, item] of Object.entries(await MCP.tools())) {
        item.inputSchema = jsonSchema(ProviderTransform.schema(model, schema))
        item.execute = async (args, opts) => {
            await ctx.ask({permission: key, ...})  // MCP 工具权限请求
            const result = await execute(args, opts)
            const truncated = await Truncate.output(textParts.join("\n\n"), {}, agent)
            return { output: truncated.content, ... }
        }
        tools[key] = item
    }
    return tools
}

// llm.ts - 阶段⑤：LLM 层面的权限过滤
async function resolveTools(input) {
    const disabled = PermissionNext.disabled(Object.keys(input.tools), input.agent.permission)
    for (const tool of Object.keys(input.tools)) {
        if (input.user.tools?.[tool] === false || disabled.has(tool)) {
            delete input.tools[tool]  // 全局 deny 的工具直接移除
        }
    }
    return input.tools
}

第九章 消息格式转换 — toModelMessages

MessageV2.toModelMessages() 将内部的 WithParts[] 消息结构转换为 AI SDK 的 ModelMessage[] 格式。这是连接 OpenCode 持久化层和 LLM API 的桥梁。

用户消息转换

// 用户消息：TextPart → text part | FilePart → file/media part | Compaction → "What did we do so far?"
if (msg.info.role === "user") {
    for (const part of msg.parts) {
        if (part.type === "text" && !part.ignored)
            userMessage.parts.push({ type: "text", text: part.text })
        if (part.type === "file" && part.mime !== "text/plain")
            userMessage.parts.push({ type: "file", url: part.url, mediaType: part.mime })
        if (part.type === "compaction")
            userMessage.parts.push({ type: "text", text: "What did we do so far?" })
        if (part.type === "subtask")
            userMessage.parts.push({ type: "text", text: "The following tool was executed by the user" })
    }
}

助手消息转换

// 助手消息：TextPart → text | ToolPart → tool-{name} | ReasoningPart → reasoning
if (msg.info.role === "assistant") {
    for (const part of msg.parts) {
        if (part.type === "text")
            assistantMessage.parts.push({ type: "text", text: part.text })
        if (part.type === "tool") {
            if (part.state.status === "completed") {
                const outputText = part.state.time.compacted
                    ? "[Old tool result content cleared]"  // 被裁剪的旧工具输出
                    : part.state.output
                assistantMessage.parts.push({
                    type: "tool-" + part.tool,
                    state: "output-available",
                    toolCallId: part.callID,
                    input: part.state.input,
                    output: outputWithAttachments,
                })
            }
            if (part.state.status === "pending" || part.state.status === "running")
                // 防止悬挂的 tool_use 块（Anthropic API 要求每个 tool_use 有 tool_result）
                assistantMessage.parts.push({
                    type: "tool-" + part.tool,
                    state: "output-error",
                    errorText: "[Tool execution was interrupted]",
                })
        }
    }
}

第十章 Compaction 上下文压缩机制

10.1 溢出检测 — isOverflow()

// compaction.ts - isOverflow()
export async function isOverflow(input) {
    const config = await Config.get()
    if (config.compaction?.auto === false) return false  // 可禁用
    
    const context = input.model.limit.context
    if (context === 0) return false  // 无限制的模型不压缩
    
    const count = input.tokens.total
        || input.tokens.input + input.tokens.output
           + input.tokens.cache.read + input.tokens.cache.write
    
    const reserved = config.compaction?.reserved
        ?? Math.min(COMPACTION_BUFFER, maxOutputTokens)  // 默认 20000
    
    const usable = input.model.limit.input
        ? input.model.limit.input - reserved
        : context - maxOutputTokens
    
    return count >= usable  // 达到可用上限则触发
}

const PRUNE_MINIMUM = 20_000   // 最少要裁剪 20K token
const PRUNE_PROTECT = 40_000   // 保护最近 40K token 不被裁剪

12.1 experimental_repairToolCall — 工具名修复

当 LLM 返回的工具名有误时，AI SDK 的 repair 机制尝试修复：

// llm.ts - 工具调用修复策略
async experimental_repairToolCall(failed) {
    const lower = failed.toolCall.toolName.toLowerCase()
    
    // 策略1：大小写修复（"Read" → "read"）
    if (lower !== failed.toolCall.toolName && tools[lower]) {
        return { ...failed.toolCall, toolName: lower }
    }
    
    // 策略2：兜底到 "invalid" 工具
    return {
        ...failed.toolCall,
        input: JSON.stringify({
            tool: failed.toolCall.toolName,
            error: failed.error.message,
        }),
        toolName: "invalid",  // invalid 工具返回错误信息给模型
    }
}

12.3 ProviderTransform 多层适配

不同 Provider 对 Schema、消息格式、参数有不同要求，通过 ProviderTransform 统一处理：

  • schema()：JSON Schema 方言转换（如 OpenAI 不支持某些 Zod 特性）
  • message()：消息预处理（通过 wrapLanguageModel middleware）
  • options()：Provider 特定选项（如 Anthropic 的 thinking budget）
  • temperature/topP/topK：根据模型能力设置采样参数

第十四章 上下文工程的 12 大设计优势

1. 分层 Prompt 架构

Agent Prompt → Provider Prompt → Environment → Instructions → User System，5 层关注点分离，每层独立可替换。

2. 模型精准适配

6 套独立 prompt 模板，针对 Claude/GPT-5/GPT-4/Gemini/Trinity/Qwen 的能力特性和 API 差异分别优化。

3. 动态工具集

每轮循环重新解析工具集，支持条件注册、权限过滤、MCP 合并、Plugin 描述修改，工具集随上下文动态变化。

4. 智能压缩策略

isOverflow 精确检测 + Compaction Agent 生成结构化摘要 + Prune 保护最近 40K token，三层防线处理上下文溢出。

5. Plan/Build 双模式

Plan 模式强制只读 + 5 阶段工作流 + plan_exit 工具退出，Build 模式无缝切换并自动注入计划文件路径。

6. 消息队列包装

step > 1 时自动将用户新消息包装为 system-reminder，防止模型将中间消息误解为新任务。

7. Prompt Caching

检测 System Prompt header 是否变化，未变时拆分为 2 段结构，最大化利用 Provider 的 KV-cache 缓存。

8. 媒体兼容性适配

检测 Provider 是否支持 tool result 中的媒体附件，不支持时自动提取为独立 user 消息。

9. 工具调用修复

experimental_repairToolCall 自动修复大小写错误，兜底到 invalid 工具返回错误信息引导模型纠正。

10. 目录级指令发现

Read 工具读文件时自动发现目录链中的 AGENTS.md/CLAUDE.md，实现"读什么文件就看到什么规范"的智能关联。

11. Plugin Transform 扩展

6 个 transform 钩子覆盖上下文构建全链路，Plugin 可注入/修改 System Prompt、消息历史、工具定义、LLM 参数。

12. 悬挂 Tool Call 防护

pending/running 状态的工具调用自动转为 output-error "[Tool execution was interrupted]"，防止 Anthropic API 因缺少 tool_result 而报错。
