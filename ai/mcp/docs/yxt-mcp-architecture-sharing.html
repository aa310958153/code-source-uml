📋 一、项目背景与目标

1.1 业务背景

随着 AI 智能体的发展，传统大模型虽能根据输入生成响应，但在实际应用中仍难以基于现有业务数据与功能灵活满足用户需求，亟需 MCP 提供辅助支持。

💡 核心痛点

  • 缺乏对业务数据的有效利用：无法基于企业已有数据进行个性化输出
  • 与业务功能集成不足：难以调用系统内具体功能模块来满足用户的真实业务诉求

1.2 建设目标

🎯 本期目标

  • 构建 MCP Server 工具发布底座
  • 屏蔽底层 MCP 协议细节
  • 支持工具快速接入与发布
  • 实现用户身份认证与权限控制
  • 落地 DevOps 助手、通义灵码插件、运营助手三大场景

🔮 中长期目标

  • 构建可视化 MCP Server 管理看板
  • 提升平台在工具治理、权限控制与版本发布等方面的运营能力
  • 支撑未来多场景、多角色的智能工具协同管理需求
  • 支持集群部署

📚 二、MCP 协议核心概念

2.1 什么是 MCP？

MCP（Model Context Protocol）是由 Anthropic（Claude 开发公司）开发的开源协议，旨在让大型语言模型（LLM）能够以标准化的方式连接到外部数据源和工具。它就像 AI 应用的通用接口，帮助开发者构建更灵活、更具上下文感知能力的 AI 应用。

📌 形象比喻：MCP 被设计为一个通用接口，类似于 USB-C 端口，允许 LLM 应用以一致的方式连接到各种数据源和工具，如文件、数据库、API 等。

2.2 三大核心角色

角色 职责 典型代表
AI 模型 (LLM) 理解用户意图，决定调用哪个工具 Claude、GPT-4、通义千问
MCP Client 从 Server 发现工具，翻译格式，转发调用 Claude Desktop、Cursor IDE、通义灵码插件
MCP Server 暴露工具、资源等能力，接收调用请求并返回结果 一心数科 mcp-server-deploy

2.3 三大核心能力

🛠️ Tools（工具）

执行操作的能力，如查询商品信息、创建会议等。Client 可以 discovery 和 invoke。

📖 Resources（资源）

提供只读的上下文信息，如文件内容、数据库记录等。类似 RAG 的外挂知识库。

📝 Prompts（模板）

预定义的交互模板，标准化常见任务，提高交互效率。

🏗️ 三、整体架构设计

3.1 架构全景图

🔍 架构分层说明

  • MCP Client 层（蓝色区域）：面向用户的 Host 主机应用，包括 DevOps 助手、IDEA 通义灵码插件、一心助手运营助手
  • MCP Server 层（绿色区域）：核心服务层，提供 SSE 连接管理、JSON-RPC 协议处理、统一鉴权、工具处理器等能力
  • 业务服务层（橙色区域）：微服务集群，包括商品库存、会员营销、销售分析、考勤核查、奖惩单据等业务系统
  • 基础设施（灰色区域）：Apollo 配置中心、微服务网关、LLM 大模型等支撑组件

⚙️ 五、关键技术特性

5.1 动态工具发布机制

基于 Apollo 配置中心实现工具的动态发布与下线，无需重新编码和部署。

// Apollo 配置：mcp_resource.json { "$schema": "https://json-schema.org/draft/2020-12/schema", "type": "object", "properties": { "awardPunishmentNo": { "type": "string", "description": "奖惩单据编号，此参数需要要求用户提供" } }, "required": ["awardPunishmentNo"], "additionalProperties": false, "yxtToolName": "assist-core-toolkit-local_getAwardPunishment", "yxtToolDescription": "根据奖惩单编号查询奖惩信息", "yxtToolEnable": true, "yxtToolProvider": { "protocol": "server", "protocolMeta": { "serverName": "assist-core-toolkit-local", "serverUrl": "/c/awardPunishment/r/1.0/getAwardPunishment" }, "alarmTagIdList": [11] } }

🔑 协议字段说明

  • yxtToolName：工具唯一标识符
  • yxtToolDescription：工具描述，用于 LLM 理解何时调用此工具
  • yxtToolEnable：工具开关，false 时自动隐藏
  • protocol：协议类型（server/git/confluence/qiyewexin）
  • protocolMeta：协议元数据，定义服务地址或文档链接
  • alarmTagIdList：工具不可用时的告警标签 ID 列表

5.2 权限控制策略

授权维度 配置示例 应用场景
基于角色 targetType: "ruleId" 不同角色（店员/店长/研发）看到不同工具集
基于用户 targetType: "userId" 精细化控制特定用户的工具访问权限
基于 AppId targetType: "appId" 不同 ChatClient（DevOps/通义灵码/运营助手）差异化授权
基于场景 targetType: "scene" 不同业务场景（考勤核查/效期核查）使用不同工具组合

5.3 会话管理机制

场景一：内部 ChatClient

特点：用户通过我们提供的 Host（如一心助手小程序）通信

实现：

  • 用户与 Host 建立连接时维护 sessionId 与用户身份信息映射
  • MCP Client 与 MCP Server 只建立一个 SSE 链接
  • MCP Client 调用时传递 sessionId 给 MCP Server 做权限过滤

场景二：主流客户端

特点：IDEA、Cursor 等独立 Host 直接与 MCP Server 通信

实现：

  • 每个 HOST 都建立一个 SSE 链接
  • 在/sse 端点扩展传递授权 appId
  • 例：/sse?appId="yxt_idea_plugin"&userId=1162324

6.2 MCPServer 划分方案对比

方案一：按业务线划分多 MCP Server

优点：

  • 各业务线服务互不干扰
  • 可独立部署、扩容、升级
  • 更容易实现访问控制和数据隔离

缺点：

  • 多实例导致配置、监控、日志难以统一管理
  • 工具组合、策略变更需逐个下发
  • 小业务线可能造成资源浪费

方案二：一个 MCP Server 统一管理 ✅推荐

优点：

  • 所有业务线可在同一平台进行工具下发、配置管理
  • 日志、监控、告警、发布等可集中处理
  • 支持跨业务线的工具组合与流程调用

缺点：

  • 单一故障点风险（可通过集群解决）

🌐 七、集群部署扩展方案

7.1 集群模式挑战

在单机模式下，MCP Client 与 MCP Server 建立 SSE 长连接后，会话信息存储在内存中。但在集群模式下，当请求被负载均衡到不同的 Server 实例时，会出现"找不到物理连接"的 404 错误。

7.2 解决方案

✅ 原生协议广播转发方案

  • 核心思路：无需新增中间件，利用 JSON-RPC 协议进行广播转发
  • 实现步骤：
    1. MCP Client 请求到达任意 Server 实例
    2. Server 检查本地会话池是否存在对应逻辑连接的物理 Session
    3. 若存在，直接处理并返回结果
    4. 若不存在，通过 Nacos 获取所有 Server 服务列表
    5. 使用 JSON-RPC 协议广播推送到所有 Server 实例
    6. 持有该 Session 的 Server 响应请求并返回结果
  • 优势：无需引入 Redis 等中间件，降低系统复杂度，利用原生协议实现轻量级会话共享
