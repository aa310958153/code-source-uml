🔌 Model Context Protocol (MCP)

从概念、架构、协议格式到裸协议实现，彻底搞懂 MCP 的本质

一、MCP 是什么？

💡 一句话概括：
MCP 定义了一套统一的接口规范，让 AI 模型能以标准化的方式发现并调用外部能力。

为什么需要 MCP？

MCP 被比作 "AI 的 USB-C 接口" —— USB-C 让所有设备用同一种接口连接外设，MCP 让所有 AI 应用用同一种协议连接工具。

二、MCP 架构：三个核心角色

🧠 AI 模型

决策者

负责理解用户意图、决定调用哪个工具

如 Claude、GPT

🌉 MCP Client

桥梁/中间人

- 从 Server 发现工具
- 翻译成 AI 能理解的格式
- AI 要调用时转发给 Server

⚙️ MCP Server

能力提供者

- 暴露工具、资源等能力
- 接收调用请求并返回结果

⚠️ 重要：AI 不直接连 Server，Client 是中间人！

三、三大核心能力

能力 类型 用途 示例
Tools（工具） POST 端点 执行操作，有副作用 发邮件、查天气、写文件
Resources（资源） GET 端点 提供数据，只读 读取文件、查询数据库记录
Prompts（提示词） 模板 预定义的交互模板 代码审查模板、翻译模板
💡 提示：Tools 是最常用的，大多数 MCP Server 只实现 Tools 就够了。

四、完整交互流程：三个阶段

第一阶段：握手 🤝
Client 和 Server 先"认识一下"，交换各自的能力
第二阶段：发现 🔍
Client 根据握手结果，去查询具体有哪些工具可用
第三阶段：调用 🚀
用户提问，AI 分析后决定调用某个工具，Client 转发执行

三阶段完整流程图

sequenceDiagram participant U as 用户 participant AI as AI 模型 participant C as MCP Client participant S as MCP Server Note over AI,S: 第一阶段：握手 🤝 C->>S: initialize (我是谁) S-->>C: 返回能力声明 (tools: {}) C->>S: notifications/initialized Note over AI,S: 第二阶段：发现 🔍 C->>S: tools/list (你有哪些工具？) S-->>C: 返回工具列表 C->>AI: 转换工具格式给 AI Note over AI,S: 第三阶段：调用 🚀 U->>AI: "北京天气怎么样？" AI->>AI: 分析：调用 get_weather(北京) AI->>C: 决定调用工具 C->>S: tools/call (name=get_weather, city=北京) S-->>C: 返回结果 (晴天 25°C) C->>AI: 传递结果 AI->>U: "北京今天晴天 25°C" Note over AI,S: 后续调用 U->>AI: "上海呢？" AI->>C: 调用 get_weather(上海) C->>S: tools/call (上海) S-->>C: 返回 (多云 22°C) AI->>U: "上海多云 22°C"

五、协议格式：一个端点 + JSON-RPC

🎯 核心机制：
所有请求都发到同一个地方：POST /mcp
靠消息体里的 method 字段路由

协议路由机制

graph LR A[所有请求] --> B[POST /mcp] B --> C{method 字段} C -->|"initialize"| D[handle_initialize] C -->|"tools/list"| E[handle_tools_list] C -->|"tools/call"| F[handle_tools_call] C -->|"notifications/*"| G[通知处理] style B fill:#667eea,color:white style C fill:#f59e0b,color:white

通用信封格式

{ "jsonrpc": "2.0", // 固定版本 "id": 1, // 请求 ID（通知无 id） "method": "xxx", // 方法名 "params": {} // 参数 / result: 结果 }

核心协议消息示例

// 请求 { "jsonrpc": "2.0", "id": 1, "method": "initialize", "params": { "protocolVersion": "2025-03-26", "capabilities": {}, "clientInfo": {"name": "my-client", "version": "1.0"} } } // 响应 { "jsonrpc": "2.0", "id": 1, "result": { "protocolVersion": "2025-03-26", "capabilities": {"tools": {}}, "serverInfo": {"name": "my-server", "version": "1.0"} } }
// 请求 { "jsonrpc": "2.0", "id": 2, "method": "tools/list", "params": {} } // 响应 { "jsonrpc": "2.0", "id": 2, "result": { "tools": [{ "name": "get_weather", "description": "获取指定城市的天气", "inputSchema": { "type": "object", "properties": { "city": {"type": "string", "description": "城市名称"} }, "required": ["city"] } }] } }
// 请求 { "jsonrpc": "2.0", "id": 3, "method": "tools/call", "params": { "name": "get_weather", "arguments": {"city": "北京"} } } // 响应 { "jsonrpc": "2.0", "id": 3, "result": { "content": [{"type": "text", "text": "晴天 25°C"}] } }

六、补充：JSON-RPC 2.0 标准

📚 JSON-RPC 2.0 是一个已有的开放标准，和 MCP 无关。
MCP 聪明地借用了这个成熟标准，只需要定义具体的 method 有哪些，以及它们的 params 和 result 格式。

七、总结回顾

MCP 核心要点总览

mindmap root((MCP)) 核心价值 标准化接口 解耦 AI 与工具 可扩展 互操作性 三个角色 AI 模型 Client 中间人 Server 执行者 三个阶段 握手 发现 调用 协议格式 JSON-RPC 2.0 POST /mcp method 路由 三大能力 Tools 最常用 Resources Prompts
🎯 关键要点回顾：

✅ MCP = AI 的 USB-C 接口
✅ 三个角色：AI 模型 → Client（中间人）→ Server
✅ 三个阶段：握手 → 发现 → 调用
✅ 一个端点：所有请求发到 /mcp，靠 method 字段路由
✅ 信封格式：JSON-RPC 2.0
✅ 三大能力：Tools（最常用）、Resources、Prompts

学习路线建议

1️⃣ 理解协议本质
先搞懂 JSON-RPC 和三个阶段的交互
2️⃣ 手写实现
用最原始的 HTTP 代码实现 Server 和 Client，看清协议的本质
3️⃣ 使用 SDK
理解底层后再用 SDK，就知道它帮你封装了什么
