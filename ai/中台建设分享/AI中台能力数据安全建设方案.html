AI中台能力数据安全建设方案

面向智能体全链路的数据行级 + 列级权限控制体系

核心目标

在AI中台四大基础设施（AI-BUS、MCP管理平台、WrenAI问数智能体、仓颉平台）的调用链路中，建设统一、标准化的数据安全防护体系，覆盖数据行级（能看哪些数据）和数据列级（能看哪些字段）两个维度的权限控制，保障数据在AI场景下的安全合规使用。

二、安全挑战与问题拆解

挑战一：行级越权 WrenAI问数 — 跨组织越权查询

🎯 场景：用户授权范围仅有"广西公司"，但通过自然语言问出"四川公司昨日销售如何"

⚠️ 风险：WrenAI直接生成无过滤条件的SQL，导致用户看到超出授权范围的数据行

挑战二：行级越权 仓颉接口 — 跨用户越权查询

🎯 场景：用户112345通过智能体询问"帮我查询112346积分"，访问他人数据

⚠️ 风险：仓颉接口无身份校验，任何入参直接穿透查询，造成水平越权

挑战三：列级泄露 敏感字段无差异化控制

🎯 场景：订单查询接口返回客户电话，客服只需看订单信息，客服主管才可看用户电话

⚠️ 风险：所有调用者获取全量字段，敏感信息无差异化保护

四、链路一：AI-BUS → 智能体 → MCP → 仓颉接口

设计思路：将权限控制标准化沉淀到仓颉平台，作为接口发布的一等配置项。用户在定义SQL、输入字段、响应字段的同时，一站式完成权限规则配置，由仓颉统一执行行/列权限过滤。

4.1 数据行级权限 — 标准化 + SPI扩展

标准化权限执行机制

权限类型 勾选配置 系统变量 SQL层效果
组织授权 勾选"启用组织权限" auth_org_codes = [用户授权组织] WHERE org_code IN (授权组织列表)
RBAC角色 勾选"启用角色权限" auth_roles = [用户角色列表] 按角色对应数据范围过滤
仅个人数据 勾选"仅本人可见" auth_emp = 当前用户工号 WHERE emp_code = 当前用户
SPI扩展 配置SPI接口地址 auth_custom = SPI返回的Map 业务自定义SQL条件填充
SPI扩展说明：当集团标准化权限无法覆盖业务特殊场景时，仓颉支持配置SPI接口。业务系统接收用户基本信息，自主判断后通过Map返回权限参数（如：是否VIP会员、可见订单类型等），这些参数会导入SQL渲染层供接口发布者自行填充使用。

4.2 数据列级权限 — 响应字段可见性配置

字段可见性配置规则

配置维度 说明 判断逻辑
角色可见 指定哪些角色可以看到该字段 用户角色 ∩ 配置角色 ≠ 空集 → 可见
组织可见 指定哪些组织可以看到该字段 用户组织 ∈ 配置组织范围 → 可见
人员可见 指定具体哪些人可以看到该字段 用户ID ∈ 可见人员列表 → 可见
SPI判断 调用业务SPI接口动态判断 SPI返回 true → 可见，false → 隐藏

五、链路二：AI-BUS → WrenAI问数智能体

5.1 数据列级权限 — DDL动态裁剪

核心思路：在WrenAI的Model维护时配置每个字段的可见性规则。当用户发起问数时，向量化给到模型的DDL会动态过滤掉用户权限范围外的字段。模型"看不到"这些字段，自然就不会在SQL中使用，从源头实现列级控制。

5.2 数据行级权限 — Query改写 + 实体抽取

核心思路：不同表有不同的权限实体（组织编号、门店编号等），有些表甚至没有直接的权限实体。在AI-BUS层通过Query改写和实体抽取两步策略，确保生成的SQL始终带有正确的权限过滤条件。

Query改写场景对比

场景 用户Query 用户权限 处理方式 最终效果
场景A "今日销售额多少" 仅门店A003 自动改写，注入权限实体 "门店A003今日销售额多少"
场景A "今日订单明细有多少条" 仅广西公司 自动改写，注入组织范围 "广西公司今日订单明细有多少条"
场景B✓ "广西公司昨日销售如何" 广西公司 抽取实体→校验通过→放行 原样传入WrenAI
场景B✗ "四川公司昨日销售如何" 仅广西公司 抽取实体→校验失败→拒绝 "抱歉，您无权查询四川公司数据"
全局权限 "全国今日销售总额" 集团权限 全局权限用户不做改写 原样传入WrenAI

无权限实体表的处理策略

问题：有些表（如订单明细表）本身没有组织编号等权限字段，但用户问数范围可能涉及这些表。
策略：非集团权限用户询问无权限实体表时，通过关联父表（如订单主表）的权限字段进行约束，或在Query改写时强制关联权限维度。
表类型 权限实体 处理策略
有权限字段的表（订单表） org_code 组织编号 直接在Query中注入组织过滤
无权限字段的表（订单明细） 无 通过血缘关系关联父表权限字段过滤
全局汇总表 无 仅集团权限用户可查询，非集团用户拒绝

六、方案评估

✅ 方案优势

  • 标准化程度高 — 仓颉侧将行/列权限沉淀为配置项，开箱即用
  • 扩展性强 — SPI机制支持业务方自主接入复杂权限逻辑
  • 双链路全覆盖 — 仓颉+WrenAI两条主链路均有完整方案
  • 用户无感 — 权限过滤对终端用户透明，不影响交互体验
  • 源头控制 — WrenAI通过DDL裁剪从模型层面杜绝越权字段出现

⚠️ 需关注的风险

  • Query改写准确性 — 自然语言理解存在歧义，需持续优化
  • SPI性能开销 — 频繁调用业务SPI可能影响响应速度
  • 权限配置复杂度 — 字段级配置较细，需提供友好的配置界面
  • 缓存一致性 — 权限变更需要及时同步，避免延迟导致越权
  • 血缘关系维护 — 无权限字段表需维护表间关联关系

优化建议

优化方向 具体措施 预期效果
配置体验 提供可视化权限配置向导 + 权限预览功能 降低配置门槛
性能优化 SPI结果缓存 + 权限数据多级缓存（本地5min/Redis30min） 减少实时调用开销
准确性保障 Query改写结果可配置人工确认 + 持续训练优化 减少误改写/漏改写

方案总结

本方案围绕AI中台双链路（AI-BUS→MCP→仓颉 / AI-BUS→WrenAI）建立了完整的数据安全防护体系：

  • 仓颉侧：通过标准化权限配置 + 系统变量注入 + SQL动态过滤 + 字段裁剪，实现行/列双维度控制
  • WrenAI侧：通过DDL动态裁剪（列级）+ Query改写 & 实体抽取（行级），从模型源头和用户输入两个环节双重保障
  • 扩展层：SPI机制为业务方提供灵活的自定义权限接入能力

方案兼顾标准化（集团通用权限开箱即用）与灵活性（SPI支持业务自定义），建议按四阶段五个月逐步落地。
